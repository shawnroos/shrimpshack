## Competing Theories

### E004 / E007: Concurrency Caps

#### Theory A: Hardware-derived concurrency [Parameter is wrong]

The hardcoded value of 4 is a guess. Optimal concurrency should be derived from the hardware profile's `memory_gb` and the per-experiment resource footprint. Each Claude Code agent + build process consumes a measurable amount of RAM and CPU. The cap should be `floor(available_memory / per_experiment_memory)` or similar.

**Prediction:** Machines with 16GB should run fewer parallel experiments than machines with 64GB. A formula like `min(floor(memory_gb / 8), cpu_cores / 2)` would produce values ranging from 2 (16GB) to 8 (64GB), meaningfully different from the static 4.

**If confirmed:** Replace the hardcoded 4 with a derived value in nerd.md auto-init, computed from hardware-profile.yaml at project init time. E007 should similarly be derived and written to global-queue.yaml during nerd-setup.

#### Theory B: Concurrency is Claude API-bound, not hardware-bound [Model is wrong]

The bottleneck is not local hardware but Claude API rate limits and context-window throughput. Each experiment agent makes many API calls. Beyond 2-3 parallel agents, API throttling or queue depth at the provider becomes the limiting factor, making local resource headroom irrelevant.

**Prediction:** Throughput (experiments completed per hour) would plateau or degrade beyond 2-3 parallel experiments regardless of local hardware capacity. The gain from 2 to 4 parallel is less than 50% (sublinear). The gain from 4 to 8 is near zero or negative.

**If confirmed:** The cap should be set conservatively (2-3) with a note that it is API-bound. Hardware derivation would be misleading. Instead, the cap should be calibrated empirically during nerd-setup by timing parallel agent launches.

#### Theory C: The cap does not matter because experiments are sequential in practice [Feature is unnecessary]

Looking at the nerd.md pipeline, Phase 5.3 merges experiments sequentially as they complete (`git merge nerd/{entry.id} --no-edit` followed by test). If merge conflicts or test failures are common, experiments effectively serialize at the merge gate. The parallelism cap only affects the *start* of experiments, not their *completion* rate.

**Prediction:** Even with cap=8, the wall-clock time from first experiment start to last merge would not improve significantly over cap=2, because the serial merge-and-test step dominates.

**If confirmed:** Parallelism cap is cosmetic. The real optimization is merge strategy (e.g., parallel-safe merge queues, or batched merges). Alternatively, if experiments are independent and don't need sequential merging, the architecture should change.

### E005 / E006: Retry Policy

#### Theory D: 3 retries x 30s is the right order of magnitude [Parameter tuning]

The retry policy is a standard exponential-backoff-style pattern. The 90s total recovery window covers most transient failures (network blips, temporary API unavailability, file locks). The values are reasonable defaults.

**Prediction:** Analyzing common Claude CLI failure modes, most transient errors resolve within 30s. A 90s window covers 95%+ of recoverable failures. Increasing beyond 3 retries has diminishing returns because failures lasting >90s are likely permanent (auth expired, disk full, etc.).

**If confirmed:** Keep current values. Document the rationale.

#### Theory E: Fixed delay should be exponential backoff [Model is wrong]

A flat 30s delay between retries is suboptimal. Transient failures cluster: if the first retry at 30s fails, the second at 60s is also likely to fail for the same reason. Exponential backoff (e.g., 15s, 60s, 240s) would better match the bimodal distribution of failure durations (either resolves in <15s or takes minutes).

**Prediction:** If we log overnight failures, we would see that retry #2 (at 60s total) rarely succeeds when retry #1 (at 30s) failed. The success pattern would be bimodal: either retry #1 works, or none of the 3 work.

**If confirmed:** Change to exponential backoff: `delay = base_delay * (2 ^ attempt)` with base_delay=15s. Total window becomes 15 + 30 + 60 = 105s, slightly longer but better distributed. Or simpler: 10s, 30s, 120s.

#### Theory F: Retries are harmful in scheduled mode [Feature is unnecessary]

In an overnight window with a fixed capacity budget, retrying a failing experiment wastes time that could be spent on the next experiment. The scheduled runner should fail fast, log the failure, and move to the next queued experiment. The retry logic makes sense for interactive use but is counterproductive in batch mode.

**Prediction:** In a window with 10 queued experiments and a capacity of 8, a single experiment that fails all 3 retries wastes 90s. With 10% failure rate, that is 9 failed experiments = 810s = 13.5 minutes wasted. More importantly, if the failure is systemic (e.g., API down), all experiments retry fruitlessly, wasting the entire window.

**If confirmed:** In scheduled mode, use max_attempts=1 (or max_attempts=2 with a short 10s delay). Redirect effort to queuing the next experiment. Add circuit-breaker logic: if 3 consecutive experiments fail, pause and log rather than burning through the queue.

### E006: Retry Sleep Delay (specifically)

#### Theory G: 30s is arbitrary and too short for the most common failure [Parameter tuning]

The most common transient failure for Claude CLI is likely API rate limiting, which typically has a backoff window of 60s+. A 30s retry would hit the same rate limit wall.

**Prediction:** Claude API rate limit headers (if exposed) show retry-after values typically >30s. Retry #1 at 30s fails more often than retry #1 at 60s.

**If confirmed:** Increase base delay to 60s, or read rate-limit headers from Claude's error output and set delay dynamically.

#### Theory H: The delay should be zero with jitter [Architecture is the bottleneck]

If multiple projects are scheduled simultaneously, their retry delays are synchronized (all sleep 30s at the same time after a shared outage). This creates thundering-herd retries. A randomized jitter (0-60s uniform random) would desynchronize retries.

**Prediction:** Multi-project overnight runs show correlated failures and correlated retries in the logs.

**If confirmed:** Replace `sleep 30` with `sleep $((RANDOM % 60 + 10))` to add jitter.

### E008: Per-Project Fairness Cap

#### Theory I: max_per_project=2 is correct for fairness [Parameter tuning]

With a global cap of 4 and the typical case of 2-3 projects, giving each project 2 slots ensures no single project monopolizes the window. The round-robin within priority tiers (documented in nerd-schedule.md:80) provides additional fairness.

**Prediction:** With 3 projects of equal backlog size and max_per_project=2 (global=4), each project gets served within the first 2 rounds. No project waits more than 2x the average.

**If confirmed:** Keep E008=2 when E007=4. Document the constraint: E008 should be `ceil(E007 / expected_project_count)`.

#### Theory J: Per-project cap should be dynamic based on queue state [Model is wrong]

A static cap of 2 is wasteful when only 1 project is scheduled. If project A has 20 experiments and no other projects are queued, it should get all 4 slots. The cap should be: `max(2, E007 - active_experiments_from_other_projects)`.

**Prediction:** In single-project overnight runs (the common case for solo developers), half the capacity is wasted because max_per_project=2 < max_codebase_experiments=4.

**If confirmed:** Replace static cap with dynamic: `min(project_backlog_size, max(floor(E007/active_project_count), 2))`. The floor of 2 guarantees fairness when multiple projects compete.

#### Theory K: Fairness is irrelevant for solo developers [Feature is unnecessary]

The multi-project scheduling system is designed for a use case (multiple active projects on one machine) that most users will never have. For a solo developer with one active project, E007 and E008 are redundant -- only E004 matters.

**Prediction:** Usage data (if collected) would show 90%+ of scheduled runs involve a single project.

**If confirmed:** Simplify: when only 1 project is in the global queue, ignore E008 and let E004 (or the hardware-derived equivalent) be the only cap. Keep E008 logic for multi-project case but don't optimize for it.

### E009: Agent Overhead Adder

#### Theory L: 60s overhead is a reasonable estimate [Parameter tuning]

Agent overhead includes: spawning Claude CLI, loading context, reading the plan, setting up the worktree environment, and writing results. 60s is a reasonable middle estimate.

**Prediction:** Measuring actual agent overhead from DAG node timestamps (plan_read_time to first_build_command) would show a distribution centered around 45-90s, with 60s near the median.

**If confirmed:** Keep 60s. Consider tightening to the measured median from production DAG data.

#### Theory M: Overhead is dominated by context loading and should scale with project size [Model is wrong]

The 60s constant assumes fixed overhead, but Claude agent startup time scales with context size. A project with a 200-line plan and 50 files has different overhead than one with a 2000-line plan and 500 files. A linear model `overhead = base + k * context_tokens` would fit better.

**Prediction:** Large projects show overhead >120s while small projects show <30s. The variance around 60s is high (CV > 0.5).

**If confirmed:** Replace constant with `overhead = 30 + (context_tokens / 10000) * 15` or similar, calibrated from DAG data. Or simpler: add a `project_size` multiplier to the hardware profile (small=30s, medium=60s, large=120s).

#### Theory N: The overhead should be measured and self-corrected from DAG data [Data is the bottleneck]

Rather than guessing, the system should measure actual overhead from completed experiments and update the constant. The DAG already stores timing data for each node. A running average of `actual_total_time - (build_time + test_time + experiment_time)` gives empirical overhead.

**Prediction:** After 10+ completed experiments, the empirical overhead would differ from 60s by more than 20% (either consistently higher or lower), and using the empirical value would improve capacity planning accuracy by reducing schedule over/under-commitment.

**If confirmed:** Add a feedback loop: after each batch, compute `mean_overhead = mean(actual_time - predicted_work_time)` from the last N experiments. Write to hardware-profile.yaml as `measured_agent_overhead_seconds`. Use this in the capacity formula instead of the hardcoded 60. Fall back to 60 for cold-start (no history).

## Analysis

### 1. Should E004 and E007 be derived from hardware profile?

**Current state:** E004 (interactive cap) is a static default written to `.claude/nerd.local.md` during auto-init. E007 (scheduled global cap) is hardcoded in the nerd-schedule.md concurrency rules. Neither reads from `hardware-profile.yaml`.

**Hardware profile contains:** `memory_gb`, `experiments_per_hour`, `build_time_seconds`, `test_time_seconds`, `chip` identifier.

**Analysis:**

The hardware profile already has the data needed to derive a sensible concurrency cap. The key constraint is memory. Each parallel experiment requires:
- One Claude CLI agent process (~500MB-1GB RSS from Node.js + context)
- One build process (language-dependent: Rust `cargo build` can peak at 2-4GB; TypeScript/Python minimal)
- Worktree disk space (minimal, CoW on APFS)

A derivation formula:

```
available_memory = memory_gb - 4  # reserve 4GB for OS + other apps
per_experiment_memory = 2  # conservative: 1GB agent + 1GB build headroom
max_parallel = floor(available_memory / per_experiment_memory)
max_parallel = clamp(max_parallel, 1, 8)  # never <1, and >8 hits API limits
```

| memory_gb | available | max_parallel (derived) | current static |
|-----------|-----------|----------------------|----------------|
| 8         | 4         | 2                    | 4 (too high)   |
| 16        | 12        | 6                    | 4 (too low)    |
| 32        | 28        | 8 (clamped)          | 4 (too low)    |
| 64        | 60        | 8 (clamped)          | 4 (too low)    |

However, Theory B (API-bound) suggests the upper clamp might be 3-4 in practice. The correct approach is:

1. Derive the hardware ceiling from memory
2. Apply the API-bound ceiling (from calibration or conservative default of 4)
3. Take the minimum

**Recommendation:** Derive E004 from hardware profile during auto-init. Derive E007 as `min(hardware_ceiling, api_ceiling, 8)` during nerd-setup. Write both as computed values with the formula documented.

### 2. Retry count x delay = total recovery window (E005 x E006)

**Current:** 3 retries x 30s = 90s total maximum recovery window.

**Failure mode analysis:**

| Failure Type                | Typical Duration | Recoverable? | Optimal Response      |
|-----------------------------|-----------------|-------------- |-----------------------|
| Claude API rate limit       | 60-120s         | Yes           | Wait + retry          |
| Network blip               | 1-10s           | Yes           | Immediate retry       |
| Auth token expired          | Permanent       | No            | Fail fast             |
| Disk full                   | Permanent       | No            | Fail fast             |
| Claude API outage           | 5-60 min        | Eventually    | Circuit breaker       |
| Git lock file               | 1-5s            | Yes           | Short retry           |

The current flat 30s x 3 pattern:
- Wastes 30s for failures that resolve in <5s (network, git locks)
- Is too short for API rate limits (often 60s+)
- Burns all retries identically for permanent failures

**Recommendation:** Exponential backoff with jitter:
```bash
delays=(10 30 120)  # total window = 160s
for attempt in 0 1 2; do
    # ... run command ...
    if [ $exit_code -ne 0 ] && [ $attempt -lt 2 ]; then
        jitter=$((RANDOM % ${delays[$attempt]} / 2))
        sleep $(( ${delays[$attempt]} + jitter ))
    fi
done
```

In scheduled mode, add circuit-breaker logic: if 2 consecutive experiments fail on retry, pause the queue and log a warning rather than burning through remaining experiments.

### 3. Can E009 (60s overhead) be measured empirically?

**Current:** Hardcoded constant `+ 60` in the capacity formula at nerd-schedule.md:53.

**DAG data available:** Each experiment node in the DAG stores timing data. The overhead is:
```
overhead = total_wall_time - (build_time + test_time + analysis_time)
```

**Bootstrap problem:** On first run, there is no DAG data. The 60s default is needed as a cold-start value.

**Recommendation:** Implement a self-correcting feedback loop:

1. **Cold start:** Use 60s (current default).
2. **After first batch:** Compute `measured_overhead = median(actual_overhead)` from DAG nodes marked `completed`. Write to `hardware-profile.yaml` as `measured_agent_overhead_seconds`.
3. **Subsequent runs:** Capacity formula reads `measured_agent_overhead_seconds` if present, falls back to 60s.
4. **Drift detection:** If measured overhead deviates from stored value by >25%, update the stored value and log: "Agent overhead re-calibrated: was {old}s, now {new}s."

The capacity formula in nerd-schedule.md becomes:
```
overhead = measured_agent_overhead_seconds || 60
adjusted_experiments_per_hour = 60 / ((effective_build_time + test_time_seconds + overhead) / 60)
```

### 4. E007/E008 interaction for fairness

**Current:** E007=4 global, E008=2 per-project. Round-robin within priority tiers.

**Fairness model:**

With N projects and global cap G=4, per-project cap P=2:

| Projects | Slots per project | Utilization | Fair? |
|----------|-------------------|-------------|-------|
| 1        | 2 (capped by P)   | 50%         | Wasteful |
| 2        | 2 each            | 100%        | Fair  |
| 3        | ~1.33 each (round-robin) | 100% | Fair  |
| 4+       | 1 each            | 100%        | Fair but slow |

The single-project case is the clear problem. Half the capacity is wasted.

**Recommendation:** Dynamic per-project cap:
```
effective_per_project = min(P, max(floor(G / active_project_count), 2))
```

When only 1 project is active, `effective_per_project = min(2, max(4, 2)) = 2` -- still capped at P=2. This is wrong. The formula should be:

```
effective_per_project = if active_project_count == 1 then G else min(P, ceil(G / active_project_count))
```

| Projects | effective_per_project | Utilization |
|----------|-----------------------|-------------|
| 1        | 4 (= G)              | 100%        |
| 2        | 2                     | 100%        |
| 3        | 2 (ceil(4/3))         | 100%        |
| 4+       | 1                     | 100%        |

This eliminates wasted capacity in the common single-project case while maintaining fairness.

## Specific Recommendations

### E004: max_parallel_experiments

**Recommendation: Derive from hardware profile. Change default from 4 to computed value.**

In `nerd.md:47`, replace the static `max_parallel_experiments: 4` with a computed value during auto-init:

```yaml
# In nerd.md auto-init logic:
# Read hardware profile
memory_gb=$(grep memory_gb ~/.claude/plugins/nerd/hardware-profile.yaml | awk '{print $2}')
available=$((memory_gb - 4))
hw_cap=$((available / 2))
# Clamp to [1, 6] -- 6 not 8 because interactive mode shares the machine with user
max_parallel=$(( hw_cap < 1 ? 1 : (hw_cap > 6 ? 6 : hw_cap) ))
```

**Rationale:** 16GB machines currently over-commit at 4 (only 12GB available after OS, divided by ~2-3GB per experiment = 4-6 safe slots, but user is also active). 32GB+ machines under-commit. The upper clamp of 6 (not 8) accounts for interactive mode where the user is also working.

**Fallback:** If no hardware profile exists, keep 4 as the default.

### E005: max_attempts (retry count)

**Recommendation: Keep at 3 for interactive mode. Reduce to 2 for scheduled mode with circuit breaker.**

The retry count itself is reasonable. The change is behavioral:

```bash
# In scheduled-run.sh:
max_attempts=2  # was 3; fail faster in batch mode
consecutive_failures=0
max_consecutive=3  # circuit breaker

# After each experiment failure:
consecutive_failures=$((consecutive_failures + 1))
if [ "$consecutive_failures" -ge "$max_consecutive" ]; then
    log "CIRCUIT BREAKER: $max_consecutive consecutive failures. Pausing queue."
    break
fi
# On success: consecutive_failures=0
```

**Rationale:** In batch/scheduled mode, wasting retries on a systemic issue (API down, auth expired) burns the entire overnight window. Fail fast, move on, and circuit-break if the problem is persistent.

### E006: Retry sleep delay

**Recommendation: Replace flat 30s with exponential backoff + jitter.**

```bash
# Replace: sleep 30
# With:
base_delays=(10 45)  # only 2 retries in scheduled mode
delay=${base_delays[$((attempt - 1))]}
jitter=$((RANDOM % (delay / 2 + 1)))
actual_delay=$((delay + jitter))
log "Retrying in ${actual_delay}s (base: ${delay}s + jitter: ${jitter}s)"
sleep $actual_delay
```

**Rationale:** First retry at 10-15s catches transient glitches. Second retry at 45-67s catches rate limits. Total window: 55-82s, comparable to current 90s but better distributed across failure modes. Jitter prevents thundering herd in multi-project schedules.

**What was analyzed and why:**

4. **Self-correction for E009:** Rather than debating whether 60s is right, the recommendation makes the system self-correcting. This is the strongest pattern because it eliminates the need to re-run this analysis as Claude's performance characteristics change.

5. **API-bound ceiling (Theory B):** This is flagged as an open question that cannot be resolved analytically. It requires empirical measurement during nerd-setup (timing parallel agent launches). The recommendation is to set a conservative upper clamp (6 interactive, 8 batch) until data is available.
