# E010-E016: Scoring and Quality Gate Parameters

Seven parameters in the nerd plugin control how research findings propagate into recommendations. These are thresholds embedded in agent instructions (markdown files) that gate decisions at every stage of the research pipeline: scope resolution, loop candidate scoring, synthesis, reporting, fixture sizing, and plan quality. This combined plan analyzes all seven together because they form an interconnected decision chain -- a finding that passes through E015's headroom tiers feeds into E012's candidate scoring, which determines what gets looped, which produces verdicts that E011 decides when to synthesize, and E013 decides whether to recommend KEEP or CHANGE.

## DAG Context

No prior theories exist in the DAG. This is the first analytical pass on these parameters. All theories are generated from scratch.

## E010: Scope Size Warning Threshold

**File:** `commands/nerd-this.md:135`
**Current value:** 50 files
**Context:** When `/nerd-this` infers scope from the user's current work, it warns if the scope exceeds 50 files, suggesting the user is approaching "full codebase scan" territory. Below 50, it proceeds silently. At 0, it stops.

### Competing Theories

#### Theory A: The absolute threshold is approximately right but should be calibrated
The 50-file threshold is a reasonable heuristic, but the optimal value depends on the LLM context window and scan cost. A different absolute number (30? 75? 100?) would better balance scope coverage against signal-to-noise.
**Prediction:** Analysis of real codebase sizes will show that 50 files corresponds to a natural break point in project structure (e.g., median module size, or an inflection in context utilization efficiency).
**If confirmed:** Adjust the threshold to the empirically identified break point.

#### Theory B: The threshold should be relative, not absolute (model is wrong)
An absolute file count is the wrong model. A codebase with 500 files treats 50 as 10% -- a focused scan. A codebase with 60 files treats 50 as 83% -- nearly everything. The threshold should be a percentage of total source files (e.g., "warn when scope exceeds 30% of the codebase").
**Prediction:** User experience of "too broad" correlates with the *fraction* of the codebase scanned, not the absolute count. Projects with 40-60 total files will frequently hit the warning despite having a genuinely narrow topic, while projects with 2000 files can scan 150 files and still be well-scoped.
**If confirmed:** Replace the absolute threshold with `max(20, floor(total_source_files * 0.3))` or similar relative formula. Keep a floor to avoid warning on tiny projects.

#### Theory C: The warning is unnecessary friction (feature is unnecessary)
Users who invoke `/nerd-this` with a topic have already expressed intent. The warning interrupts flow without preventing real problems -- the actual cost of scanning 80 vs 40 files is trivial compared to the LLM agent overhead. The scope confirmation step (Phase 2) already shows the file count and lets users adjust.
**Prediction:** In practice, users who see the warning almost always proceed anyway. The warning adds a decision point that rarely changes behavior. Removing it produces no measurable degradation in scan quality or user satisfaction.
**If confirmed:** Remove the warning entirely. The Phase 2 scope confirmation already serves this purpose.

### Methodology

This is an analytical experiment -- no parameter sweep is possible since the threshold lives in markdown instructions, not executable code.

1. **Codebase size distribution analysis:** Survey real-world codebases to characterize the distribution of source file counts. Determine what fraction of projects would trigger the warning at thresholds of 20, 30, 50, 75, 100.

2. **Relative vs absolute comparison:** For each threshold candidate, compute both the absolute and relative (percentage-of-codebase) values. Identify which model produces more consistent "scope quality" classifications across projects of different sizes.

3. **Phase 2 redundancy check:** Analyze whether the scope confirmation step in Phase 2 makes the scope size warning redundant. If Phase 2 always shows file counts and allows adjustment, the warning at Phase 1 may be pure friction.

4. **Context window analysis:** Calculate the token cost of scanning N files at representative file sizes (median source file ~100 lines, ~3000 tokens). Identify the file count where context utilization exceeds 50% of the available window, leaving insufficient room for analysis.

### Acceptance Criteria

- AC1 (Theory A): If analysis shows 50 files corresponds to a natural inflection in context utilization or scan quality, KEEP with documentation of the rationale.
- AC2 (Theory B): If the relative model produces >20% fewer false positives (warnings on well-scoped scans) across a sample of project sizes, CHANGE to relative threshold.
- AC3 (Theory C): If Phase 2 provides equivalent or better scope control, CHANGE to remove the warning.

## E011: Synthesis Min Verdict Count

**File:** `agents/loop-scout.md:169`
**Current value:** 3 verdicts
**Context:** The loop-scout creates synthesis nodes in the global DAG when 3 or more verdicts in the same tag cluster reach the same conclusion. Synthesis nodes represent cross-experiment patterns (e.g., "data quality dominates threshold tuning"). The threshold also controls confidence levels: 3 = low, 3-4 = medium, 5+ = high.

### Competing Theories

#### Theory A: The threshold of 3 is too low, producing premature synthesis
Three verdicts is not enough to establish a pattern. With small numbers, coincidental agreement is likely. The threshold should be 5 to avoid polluting the DAG with spurious patterns that mislead future experiment design.
**Prediction:** At n=3, the false positive rate for synthesis claims (patterns that don't replicate when more experiments run) exceeds 30%. Raising to n=5 would reduce false patterns without missing real ones.
**If confirmed:** Raise the minimum to 5 and collapse the confidence tiers to: 5-7 = medium, 8+ = high.

#### Theory B: The threshold of 3 is correct given the sparse-data reality
Nerd projects are unlikely to accumulate many verdicts on the same topic. If the threshold is 5, most projects will never produce a synthesis node, making the entire synthesis mechanism dead code. The value of 3 balances statistical caution against practical utility.
**Prediction:** Analysis of expected verdict accumulation rates shows that typical projects produce 5-15 total verdicts across all tags. At threshold=5, fewer than 20% of projects would ever produce a synthesis node. At threshold=3, 40-60% would.
**If confirmed:** KEEP at 3, but add a decay mechanism -- synthesis nodes created from exactly 3 verdicts get `confidence: "provisional"` and are demoted if no additional supporting evidence appears within 5 subsequent experiments.

#### Theory C: Count-based thresholds are the wrong model -- use effect consistency instead (model is wrong)
The number of verdicts matters less than their consistency. Three verdicts that all say "data quality is the bottleneck" with high confidence are more meaningful than five verdicts where three weakly agree and two are ambiguous. The synthesis trigger should be based on agreement strength, not count.
**Prediction:** Simulating verdict sets with varying agreement levels shows that count alone is a poor predictor of pattern validity. An agreement-weighted metric (e.g., requiring mean verdict confidence > 0.7 AND count >= 2) would produce higher-quality synthesis nodes.
**If confirmed:** Replace the count threshold with a composite trigger: `count >= 2 AND mean_confidence >= "medium" AND all verdicts in same direction (all SUPPORTED or all REFUTED)`.

### Methodology

1. **Verdict accumulation modeling:** Estimate how many verdicts a typical nerd project produces per tag cluster over 10, 20, 50 experiments. Use the nerd plugin's own structure as a case study -- how many tag clusters exist, how many experiments per cluster?

2. **Small-sample agreement analysis:** Using combinatorics, calculate the probability that 3 randomly-tagged verdicts will share a tag cluster by chance (birthday problem analog). Compare against threshold=4 and threshold=5.

3. **Confidence tier analysis:** Evaluate whether the current tiers (3 = low, 3-4 = medium, 5+ = high) are well-calibrated. Does the jump from 3 to 5 verdicts meaningfully increase pattern reliability?

4. **Ablation:** Consider what happens if synthesis is disabled entirely. Does the DAG still function? Are synthesis nodes consumed downstream, or are they informational dead ends?

### Acceptance Criteria

- AC1 (Theory A): If birthday-problem analysis shows >25% chance of spurious 3-verdict clusters in typical project sizes, CHANGE to threshold=5.
- AC2 (Theory B): If expected verdict accumulation shows <20% of projects reaching 5 verdicts per cluster, KEEP at 3 with provisional confidence tier.
- AC3 (Theory C): If agreement-weighted triggers produce higher-quality synthesis in simulation, CHANGE to composite trigger model.

## E012: Loop Candidate Score Thresholds

**File:** `agents/loop-scout.md:105`
**Current value:** Score >= 7 = strong candidate, Score 5-6 = viable, Score < 5 = not worth looping
**Weights:** 40% headroom, 25% metric quality, 20% scope clarity, 15% theory richness
**Context:** The loop-scout scores potential loop targets on four weighted dimensions and uses threshold-based decision rules to classify candidates. This determines which findings get deep `/nerd-loop` iteration vs one-off code changes.

### Competing Theories

#### Theory A: The thresholds are reasonable but the weights are wrong
The 7/5 thresholds are defensible cut points, but the dimension weights (40/25/20/15) over-index on headroom and under-index on metric quality. A loop on a 15% improvement opportunity with a broken metric is worse than a loop on a 5% opportunity with a precise, fast eval harness.
**Prediction:** Post-hoc analysis of loop outcomes will show that metric quality is a stronger predictor of loop success than headroom. Reweighting to 30% headroom, 35% metric quality, 20% scope clarity, 15% theory richness would better rank candidates.
**If confirmed:** Adjust weights. Keep thresholds.

#### Theory B: The discrete tier system should be continuous (model is wrong)
Mapping each dimension to High/Medium/Low/Skip and then combining with fixed weights loses information. A 9.5% improvement (just under the "High" headroom cutoff) scores the same as a 3.1% improvement (just above "Medium"). A continuous scoring function would preserve gradient information and produce better rankings.
**Prediction:** At the tier boundaries (e.g., 10.1% vs 9.9% improvement), the scoring system produces discontinuous jumps that change candidate ranking. Real-world improvement values cluster near boundaries more often than expected, making the discrete model unreliable.
**If confirmed:** Replace tier-based scoring with continuous functions. E.g., headroom score = min(improvement_pct / 15, 1.0) for a linear ramp, or a sigmoid for diminishing returns.

#### Theory C: The four-dimension model is missing a critical factor -- estimated iteration cost (architecture is wrong)
Headroom, metric quality, scope clarity, and theory richness are all about the *value* of looping. But loops also have *cost* -- how many iterations to plateau, how long each iteration takes, how much human review is needed. A candidate with score 8 that requires 50 iterations at 10 minutes each is worse than a score 6 candidate that plateaus in 3 quick iterations.
**Prediction:** Adding an "iteration efficiency" dimension (estimated iterations to plateau, time per iteration) would change the top-ranked candidate in >30% of real scoring scenarios.
**If confirmed:** Add a fifth dimension: iteration efficiency (weight: 15%, redistributed from headroom dropping to 30%). The loop-scout already estimates "iterations to plateau" in its output -- wire this into the score.

### Methodology

1. **Weight sensitivity analysis:** For each dimension weight, compute the score change when shifting +/-10%. Identify which weight changes have the largest impact on candidate ranking.

2. **Boundary analysis:** Identify the tier boundaries for each dimension. Compute what fraction of plausible real-world values fall within 1% of a boundary, making the discrete classification unstable.

3. **Missing dimension analysis:** Enumerate what factors the current model does NOT capture (iteration cost, dependency risk, staleness of evidence). For each, estimate how often including it would change the top-ranked candidate.

4. **Threshold calibration:** At the current weights, compute the score distributions for "obviously good" and "obviously bad" loop candidates. Verify that 7 and 5 cleanly separate these distributions.

### Acceptance Criteria

- AC1 (Theory A): If sensitivity analysis shows metric quality weight changes affect ranking more than headroom weight changes, CHANGE weights.
- AC2 (Theory B): If >15% of plausible values fall within 1 tier-unit of a boundary, CHANGE to continuous scoring.
- AC3 (Theory C): If adding iteration cost dimension changes the top candidate in >25% of scenarios, CHANGE to five-dimension model.

## E013: KEEP vs CHANGE Improvement Threshold

**File:** `skills/research-reporting/SKILL.md:22`
**Current value:** 1% improvement
**Context:** When an experiment finds that the best configuration differs from the current one, a CHANGE recommendation is issued only if the improvement is >= 1%. Below 1%, the recommendation is KEEP (treat the difference as noise). This is the single gate between "we found something" and "we recommend changing something."

### Competing Theories

#### Theory A: 1% is too low -- it cannot be distinguished from experimental noise
In analytical experiments on markdown-file parameters (the nerd plugin's domain), measurement precision is inherently limited. Metrics derived from LLM behavior have high variance. A 1% improvement may be within the noise floor, causing the system to recommend changes that are actually random fluctuation.
**Prediction:** Repeated runs of the same experiment with the same parameters will show result variance exceeding 1%. The noise floor is more likely 3-5%, making any improvement below that indistinguishable from measurement error.
**If confirmed:** Raise the threshold to 3% (or better, to 2x the measured standard deviation of the metric). Add a requirement that improvements must exceed the noise floor with 95% confidence.

#### Theory B: A fixed percentage threshold is the wrong model -- use statistical significance (model is wrong)
A flat 1% threshold ignores sample size, variance, and effect size. An improvement of 1% measured over 10,000 data points is highly significant. An improvement of 5% measured over 10 data points is noise. The threshold should be a statistical test (e.g., paired t-test, bootstrap confidence interval), not a percentage.
**Prediction:** There exist cases where a 0.5% improvement is statistically significant (large N, low variance) and cases where a 3% improvement is not (small N, high variance). A fixed threshold makes the wrong call in both scenarios.
**If confirmed:** Replace the percentage threshold with: "CHANGE if improvement is statistically significant (p < 0.05, or 95% bootstrap CI excludes zero) AND the effect size exceeds a practical significance threshold (e.g., 0.5%)."

#### Theory C: The threshold is correct for its purpose -- it is a conservatism gate, not a noise filter
The 1% threshold is not trying to be a statistical test. It is a conservatism heuristic: "don't recommend changes for tiny improvements because the cost of making the change (review, testing, risk) exceeds the benefit." In the nerd context, changing a markdown parameter is nearly zero-cost, so even 1% is conservative enough.
**Prediction:** The cost of changing a parameter in a markdown file is effectively zero (no compilation, no deployment, no migration). Therefore the threshold should be even lower -- any measurable improvement should trigger CHANGE, and the threshold exists only to filter rounding errors.
**If confirmed:** Lower to 0.5% or remove entirely, replacing with "CHANGE if improvement is non-zero and consistent across evaluation conditions."

#### Theory D: The threshold should vary by experiment type (architecture is wrong)
Different experiments have different noise floors. A parameter sweep on a deterministic algorithm has near-zero noise -- 0.1% is meaningful. An LLM-evaluated metric has 5-10% run-to-run variance -- 1% is noise. A single global threshold cannot serve both.
**Prediction:** Experiments in the nerd pipeline span a wide range of noise characteristics. Grouping by experiment type will show that optimal thresholds differ by 5x or more between categories.
**If confirmed:** Replace the single threshold with a per-experiment-type threshold table, or require each experiment plan to declare its noise floor and set the KEEP/CHANGE threshold at 2x the declared noise.

### Methodology

1. **Noise floor estimation:** For the class of experiments nerd runs (analytical markdown parameter analysis, LLM-driven evaluations, code metric sweeps), estimate the typical measurement variance. Use bootstrapping on synthetic data to characterize expected run-to-run variation.

2. **Cost-benefit analysis:** Enumerate the cost of making a CHANGE recommendation in the nerd context. If the cost is near-zero (editing a markdown file), the threshold can be lower. If there are downstream costs (plans that depend on the parameter, retraining required, user confusion from frequent changes), the threshold should be higher.

3. **Historical comparison:** Survey the experimental design literature for typical significance thresholds in A/B testing (typically 5% significance with >1% minimum detectable effect) and ML hyperparameter tuning (typically >0.5% improvement required).

4. **Type-stratified analysis:** Classify nerd experiments by noise profile and compute appropriate thresholds for each class.

### Acceptance Criteria

- AC1 (Theory A): If estimated noise floor for nerd experiments exceeds 1%, CHANGE to max(3%, 2 * estimated_noise_floor).
- AC2 (Theory B): If statistical significance testing is feasible for nerd experiments (sufficient data, computable variance), CHANGE to significance-based threshold.
- AC3 (Theory C): If change cost is near-zero and no downstream dependencies exist, CHANGE to 0.5% or lower.
- AC4 (Theory D): If noise floor varies by >3x across experiment types, CHANGE to per-type thresholds.

## E014: Test Fixture Max Records

**File:** `agents/lab-tech.md:297`
**Current value:** 1000 records
**Context:** When the lab-tech agent scaffolds test fixtures for experiments, it caps fixture size at 1000 records to keep iteration fast. This applies to all experiment types regardless of the data characteristics.

### Competing Theories

#### Theory A: 1000 is a reasonable default but should be stated as a guideline, not a hard cap
The 1000-record limit serves its purpose for most experiments -- fast iteration during development. But some experiments (e.g., those measuring distribution tails or rare-event detection) need more data to produce meaningful results. The cap should be a default with an override mechanism.
**Prediction:** Most experiments work fine with <=1000 records. But for experiments measuring P95+ metrics, rare-event rates, or distribution shapes, 1000 records is insufficient to produce stable measurements. These experiments need 5000-10000 records.
**If confirmed:** Keep 1000 as the default but add an `override_fixture_size` field to experiment plans. Lab-tech respects the override when specified.

#### Theory B: The cap should vary by data sparsity and metric type (model is wrong)
A fixed cap ignores data characteristics. If the phenomenon being measured occurs in 1% of records, 1000 records gives you ~10 positive examples -- too few for statistical stability. If the phenomenon occurs in 50% of records, 200 records suffices. The cap should be derived from the expected signal density.
**Prediction:** Fixture quality (measured by metric stability across bootstrap resamples) varies dramatically with signal density at a fixed size of 1000. For dense signals (>10% prevalence), 1000 is more than enough. For sparse signals (<1% prevalence), 1000 produces unstable metrics with >20% variance across resamples.
**If confirmed:** Replace the fixed cap with a formula: `min(10000, max(200, ceil(100 / expected_prevalence)))`. This ensures at least 100 positive examples in the fixture while capping at 10000 for practical reasons.

#### Theory C: Fixture size does not matter because the fixture is not the bottleneck (data is the bottleneck)
The quality of the fixture matters more than its size. A 1000-record fixture drawn from a biased sample is worse than a 200-record fixture drawn from a representative sample. The lab-tech should focus on sampling strategy (stratified, importance-weighted) rather than raw count.
**Prediction:** Varying fixture size from 200 to 5000 while holding sampling strategy constant shows diminishing returns past ~500 records for most metrics. But switching from random sampling to stratified sampling at a fixed 500 records improves metric stability more than increasing random sample size from 500 to 5000.
**If confirmed:** Reduce default to 500 records but add stratified sampling requirements. Lab-tech should ensure class balance and coverage of edge cases rather than maximizing count.

### Methodology

1. **Signal density analysis:** For the types of experiments nerd runs, characterize the expected distribution of signal density (what fraction of records are "positive" or "informative" for typical metrics).

2. **Stability analysis:** At fixture sizes of 100, 200, 500, 1000, 2000, 5000, compute metric stability (coefficient of variation across bootstrap resamples) for representative metrics at various signal densities.

3. **Sampling strategy comparison:** Compare random vs stratified sampling at fixed fixture sizes. Measure which factor (size vs strategy) has a larger impact on metric stability.

4. **Iteration speed tradeoff:** Measure the wall-clock cost of running eval harnesses at different fixture sizes. Identify the knee point where larger fixtures cause unacceptable slowdown.

### Acceptance Criteria

- AC1 (Theory A): If >20% of experiment types need more than 1000 records for stable metrics, CHANGE to a default-with-override model.
- AC2 (Theory B): If metric stability varies by >50% across signal densities at fixed size=1000, CHANGE to density-adaptive formula.
- AC3 (Theory C): If stratified sampling at n=500 matches or exceeds random sampling at n=2000 in metric stability, CHANGE to emphasize sampling strategy over size.

## E015: Headroom Tier Thresholds

**File:** `agents/loop-scout.md:41`
**Current value:** >10% = High, 3-10% = Medium, <3% = Low (implicit)
**Context:** The loop-scout uses these tiers to score improvement headroom -- the largest-weighted dimension (40%) in loop candidate scoring. An experiment showing >10% possible improvement gets "High" headroom, which strongly favors that candidate for deep iteration.

### Competing Theories

#### Theory A: The tier boundaries are reasonable but the gap between Medium and Low is too narrow
The 3% lower bound of Medium means that any experiment showing 2.9% improvement gets Low headroom, while 3.1% gets Medium. This is a 0.2 percentage point difference producing a meaningful score change at 40% weight. The tiers should have wider gaps or the boundaries should shift.
**Prediction:** The 3% boundary is particularly sensitive because it sits near the noise floor for many metrics (see E013 analysis). Small measurement variations can flip a candidate between Low and Medium, changing its overall score by ~1.3 points (0.4 weight * 3.3 tier difference on a 10-point scale).
**If confirmed:** Widen the Medium band: >15% High, 5-15% Medium, <5% Low. This creates clearer separation and puts the Low/Medium boundary above typical noise floors.

#### Theory B: The tiers should be relative to the metric's baseline, not absolute percentages (model is wrong)
A 5% improvement from 90% accuracy to 94.5% is substantial (halving the error rate from 10% to 5.5%). A 5% improvement from 20% accuracy to 21% is trivial. The headroom tiers should account for where the baseline sits.
**Prediction:** Experiments near ceiling performance (>85% on their metric) show systematically different improvement-to-value curves than experiments with low baselines (<50%). Using absolute percentage thresholds misprices headroom in both cases.
**If confirmed:** Replace absolute thresholds with relative-to-headroom-remaining: High = improvement > 30% of remaining headroom, Medium = 10-30%, Low = <10%. Example: at 90% accuracy, 3% improvement = 30% of remaining 10% headroom = High.

#### Theory C: Three tiers are insufficient -- use four or five (architecture is wrong)
The current model maps a continuous improvement percentage onto three bins plus a Skip sentinel. This loses too much information. The difference between "10.1% improvement" and "50% improvement" is enormous, but both score as "High." Adding a "Very High" tier (>25%) and splitting "Medium" into "Medium-High" (7-10%) and "Medium-Low" (3-7%) would preserve useful ranking information.
**Prediction:** Among candidates that all score "High" on headroom, the actual improvement percentages vary by 5x or more. The current system cannot distinguish a strong 12% candidate from an exceptional 40% candidate.
**If confirmed:** Add two tiers: >25% = Very High, 10-25% = High, 5-10% = Medium, <5% = Low. Or switch to continuous scoring (see E012 Theory B -- these are related).

#### Theory D: The interaction with E013's 1% threshold creates a dead zone
E013 says improvements below 1% get KEEP recommendations. E015 implicitly treats <3% as Low headroom. This creates a zone from 1-3% where experiments trigger CHANGE recommendations but score as Low headroom for future iteration. This dead zone may cause the system to recommend a change but never iterate further on it, even though 1-3% improvements could compound.
**Prediction:** Parameters that show 1-3% improvement in initial experiments get changed (per E013) but never looped on (Low headroom per E015). Some of these could yield further compounding improvements if looped, but the system systematically ignores them.
**If confirmed:** Align the thresholds: either raise E013 to 3% (matching E015's Low boundary) or add a "compounding potential" factor that evaluates whether the initial improvement suggests more is available.

### Methodology

1. **Tier boundary sensitivity:** At the current weights, compute how much a candidate's overall score changes when improvement moves across each tier boundary (2.9% to 3.1%, 9.9% to 10.1%). Compare this sensitivity to the expected measurement noise from E013 analysis.

2. **Absolute vs relative comparison:** Using representative experiments at different baseline performance levels, compare rankings produced by absolute tiers vs relative-to-remaining-headroom tiers.

3. **Information loss quantification:** For a set of hypothetical candidates with known improvement percentages, measure how much ranking information the 3-tier system preserves vs 4-tier, 5-tier, and continuous scoring.

4. **Dead zone analysis:** Map the interaction between E013 (KEEP/CHANGE threshold) and E015 (headroom tiers) to identify parameter regions where the system produces contradictory or suboptimal guidance.

### Acceptance Criteria

- AC1 (Theory A): If the 3% boundary sits within 1 standard deviation of typical noise floors, CHANGE to widen the Medium band.
- AC2 (Theory B): If relative-to-remaining-headroom scoring produces different top-3 rankings in >25% of test scenarios, CHANGE to relative model.
- AC3 (Theory C): If continuous scoring is adopted in E012 (related theory), align E015 by switching to continuous headroom scoring.
- AC4 (Theory D): If the 1-3% dead zone analysis reveals >20% of initial improvements in this range have compounding potential, CHANGE to align E013 and E015 boundaries.

## E016: Min Competing Theories

**File:** `agents/plan-reviewer.md:69`
**Current value:** 3 theories minimum
**Context:** The plan-reviewer agent requires at least 3 competing theories per experiment plan. This quality gate ensures experiments are designed to distinguish between explanations rather than just sweep parameters. The framework provides six theory types: parameter is wrong, model is wrong, feature is unnecessary, metric is wrong, data is the bottleneck, architecture is the bottleneck.

### Competing Theories

#### Theory A: Three is the right number based on cognitive science
Research on hypothesis generation shows that considering 2 alternatives is insufficient to avoid confirmation bias (Klayman & Ha, 1987), but generating more than 4-5 alternatives produces diminishing returns and "hypothesis dilution" where weak theories crowd out strong ones (Dougherty et al., 2010). Three is the sweet spot: one primary hypothesis, one structural alternative, one meta-challenge (questioning the metric or data).
**Prediction:** Examination of the six theory types shows they naturally cluster into three functional roles: (1) parameter optimization (types 1-2), (2) structural critique (types 3, 6), (3) measurement challenge (types 4-5). Requiring one from each cluster produces higher-quality experiment designs than requiring raw count.
**If confirmed:** KEEP at 3 but add a diversity requirement: at least one theory must challenge the metric, data, or architecture (not just propose different parameter values).

#### Theory B: Three is too low -- plans need 4-5 theories to avoid blind spots
The framework lists six theory types, but requiring only three means half the theory space goes unexplored. Important failure modes (especially "metric is wrong" and "data is the bottleneck") are systematically underrepresented because experimenters gravitate toward the easier "parameter is wrong" and "model is wrong" theories.
**Prediction:** Plans with exactly 3 theories will show a systematic gap: >70% will include "parameter is wrong" and >60% will include "model is wrong," but <30% will include "metric is wrong" or "data is the bottleneck." Raising to 4 theories with a diversity requirement would close this gap.
**If confirmed:** Raise to 4 minimum with at least one theory from each of these groups: (1) optimization (parameter/model), (2) simplification (feature is unnecessary), (3) foundations (metric/data/architecture).

#### Theory C: Mandating a minimum count produces low-quality filler theories (feature is unnecessary)
A rigid minimum forces plan-reviewers to generate theories even when fewer are appropriate. For simple experiments (e.g., "should this timeout be 30s or 60s?"), the third theory is often contrived -- it doesn't add analytical value and clutters the plan. Quality matters more than quantity.
**Prediction:** In plans with exactly 3 theories, the third theory will be measurably weaker: less specific predictions, vaguer "if confirmed" actions, and less connection to the actual experiment design. Removing the minimum and replacing with quality criteria (each theory must have a testable prediction that the experiment can distinguish) would produce better plans overall.
**If confirmed:** Remove the numeric minimum. Replace with: "Generate theories until you have at least one that challenges the experiment's metric or data validity. Each theory must have a testable, distinguishable prediction. Do not generate theories that the experiment cannot discriminate between."

#### Theory D: The cognitive science supports 3 but the framing matters more than the count
Charness & Levitin (2006) showed that expertise in hypothesis generation correlates with the ability to generate *disconfirming* hypotheses, not with the total count. The requirement should emphasize generating at least one theory that would *disprove* the primary hypothesis, rather than simply generating N alternatives.
**Prediction:** Plans where at least one theory is explicitly framed as a disconfirmation of the primary hypothesis produce better experiment designs (more discriminating tests, clearer acceptance criteria) than plans with more theories but no explicit disconfirmation.
**If confirmed:** Keep minimum at 3 but add: "At least one theory must predict that the primary hypothesis is wrong and specify what observation would confirm this."

### Methodology

1. **Cognitive science review:** Survey the hypothesis generation literature for empirical findings on optimal alternative counts. Key papers: Klayman & Ha (1987) on confirmation bias, Mynatt et al. (1977) on disconfirmation, Thomas et al. (2008) on hypothesis generation in scientific reasoning, Dougherty et al. (2010) on hypothesis dilution effects.

2. **Theory type distribution analysis:** For the six theory types in the framework, analyze which types are most and least likely to be generated at minimum counts of 2, 3, 4, and 5. Identify systematic blind spots.

3. **Quality vs quantity tradeoff:** Define a theory quality metric (specificity of prediction, actionability of "if confirmed" clause, discriminability from other theories). Analyze whether higher minimums dilute average quality.

4. **Disconfirmation analysis:** Review whether the current framework incentivizes disconfirming theories or whether all 3+ theories tend to be variants of the primary hypothesis ("parameter is wrong at value X" vs "parameter is wrong at value Y").

### Acceptance Criteria

- AC1 (Theory A): If cognitive science supports 3 as the sweet spot and natural clustering into three roles is confirmed, KEEP with added diversity requirement.
- AC2 (Theory B): If analysis shows >50% of plans omit metric/data/architecture theories at min=3, CHANGE to 4 with diversity gates.
- AC3 (Theory C): If quality analysis shows the marginal theory at count=3 is measurably weaker than the first two, CHANGE to quality-based requirement.
- AC4 (Theory D): If disconfirmation framing produces better experiment designs regardless of count, ADD disconfirmation requirement and keep count at 3.

## Cross-Parameter Interactions

These seven parameters do not operate in isolation. Several interactions are critical:

### E013 x E015: The Dead Zone
The KEEP/CHANGE threshold (1%) and the Low headroom boundary (3%) create a dead zone from 1-3% where changes are recommended but never iterated on. This interaction should be analyzed as a system, not as independent parameters.

**Recommendation:** Align these boundaries. Either raise E013 to match E015's lower bound, or add explicit handling for the 1-3% range in loop-scout scoring.

### E012 x E015: Headroom Feeds Scoring
E015's headroom tiers are the largest input (40% weight) to E012's candidate scoring. Any change to E015 (tier boundaries, continuous vs discrete, absolute vs relative) directly changes E012's output. These should be analyzed together.

**Recommendation:** If E012 moves to continuous scoring, E015's tier system becomes unnecessary -- replace tiers with a continuous headroom function.

### E011 x E016: Theory Count Drives Verdict Count
E016 requires 3+ theories per experiment. Each theory produces at most one verdict. E011 requires 3+ verdicts per tag cluster for synthesis. This means synthesis requires at minimum 3 experiments with overlapping tags to trigger, which is appropriate -- but if E016 is raised to 4-5 theories, the verdict volume increases and E011's threshold may need adjustment.

**Recommendation:** Analyze these as a pipeline: theory count -> verdict count -> synthesis threshold. Ensure the chain produces synthesis at realistic project scales.

### E010 x E014: Scope and Fixture Sizing
E010 gates how many files enter scope. E014 caps fixture size. If a project has 500 files and E010 scopes 80 of them, the fixtures may need more records to cover the broader scope. These parameters may need to co-vary.

**Recommendation:** Consider whether fixture size should scale with scope size, or whether they are genuinely independent.

## Ground Truth Strategy

These parameters live in markdown instructions consumed by LLM agents. There is no traditional ground truth -- no labeled dataset, no deterministic output. The "correct" threshold is the one that produces the best research outcomes, which is inherently circular (the thresholds define what "best" means).

**Mitigation strategies:**
1. **Expert judgment baseline:** Use the plan author's (Shawn's) experience with the nerd plugin to establish priors on what "good scope," "meaningful improvement," and "sufficient theories" look like.
2. **Consistency analysis:** A good threshold produces consistent decisions -- the same experiment should get the same KEEP/CHANGE recommendation regardless of minor perturbations in the data.
3. **Boundary case analysis:** Stress-test each threshold at its boundary values. If small changes in the input produce large changes in the output, the threshold is too sensitive.
4. **First-principles reasoning:** For some parameters (E016 especially), cognitive science and statistical theory provide independent constraints.

## Specific Recommendations

Based on the analysis structure above, preliminary recommendations (to be confirmed or refuted by the analytical work):

1. **E010:** Switch to relative threshold: `max(20, floor(total_source_files * 0.3))`. An absolute count of 50 is arbitrary and poorly calibrated for projects of different sizes. The relative model handles both tiny and huge projects gracefully.

2. **E011:** KEEP at 3 but add a `provisional` confidence tier for exactly-3-verdict synthesis nodes. These should be flagged for human review and automatically demoted if unsupported by subsequent experiments.

3. **E012:** Move to continuous scoring for all dimensions. The discrete tier system loses information and creates unstable boundary behavior. Keep the 7/5 thresholds as they map well to continuous scores.

4. **E013:** Raise to 3% as the default KEEP/CHANGE threshold, with a per-experiment override when the noise floor is characterized. 1% is almost certainly below the noise floor for LLM-evaluated metrics.

5. **E014:** Change to a density-adaptive formula: `min(5000, max(200, ceil(100 / expected_prevalence)))`. Add stratified sampling as a requirement alongside the size cap.

6. **E015:** If E012 adopts continuous scoring, replace tiers with a continuous headroom function. If E012 keeps tiers, widen the Medium band to 5-15% and raise High to >15%.

7. **E016:** KEEP at 3 but add two requirements: (a) at least one theory must be a disconfirmation of the primary hypothesis, (b) at least one theory must challenge the metric, data, or architecture rather than proposing a different parameter value.
