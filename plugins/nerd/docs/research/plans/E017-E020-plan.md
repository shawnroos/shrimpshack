## E017: Max Theme Count (currently 6)

**File:** `agents/context-scanner.md:104`
**Current value:** Maximum 6 themes, ideal 3-5, excess merged into "Other"
**Role:** Caps the number of research themes the context-scanner returns when clustering discovered parameters. Prevents the user from being overwhelmed by too many theme choices.

### How It Works in Context

The context-scanner scans a scoped file list, discovers tunable parameters, then clusters them into themes by functional role. Lines 100-105 define the theme size guidelines:

- Minimum 1 theme (if everything belongs together)
- Maximum 6 themes (merge smallest into "Other" if exceeded)
- Ideal 3-5 themes, each with 3-15 parameters
- Single-parameter themes get merged

The cap is a flat constant. It does not vary with scope size (number of files scanned, number of parameters found, or project complexity).

### Competing Theories

#### Theory A: The cap should scale with scope size (parameter tuning via Miller's Law)

The value 6 is a rough application of Miller's 7+/-2 law (human working memory holds ~7 items). But Miller's Law applies to short-term memory for unstructured lists. Theme selection is a structured decision with labels and descriptions -- more like a menu than a memory task. Research on menu usability (Kiger 1984, Miller 1981 -- a different Miller) suggests broader menus with 8-16 items can outperform deep hierarchies when items are well-labeled.

More importantly, the cap is flat regardless of whether the scanner finds 10 parameters across 3 files or 200 parameters across 50 files. A flat cap means:
- Small scopes: 6 themes is probably too many (3 files might only warrant 2 themes)
- Large scopes: 6 themes forces aggressive merging that may hide distinct research areas

**Prediction:** For scopes with >100 parameters, the "Other" bucket will consistently contain >30% of all parameters, indicating information loss from premature merging. For scopes with <15 parameters, the scanner will produce themes with only 1-2 parameters each.

**If confirmed:** Replace flat cap with a formula: `max_themes = clamp(ceil(sqrt(parameter_count)), 2, 10)`. This gives 2 themes for 4 parameters, 5 themes for 25, 8 themes for 64, capped at 10.

#### Theory B: The cap is irrelevant because the LLM ignores it anyway (feature is unnecessary)

The theme count guideline is a soft instruction to an LLM (the context-scanner agent uses Sonnet). LLMs notoriously drift from numeric constraints in system prompts, especially when the instruction says "ideal 3-5" but "maximum 6." The LLM may already be producing 7+ themes when the data warrants it, or collapsing to 2-3 regardless. The guideline may be cosmetic rather than functional.

**Prediction:** Running the context-scanner on 5+ diverse scopes of varying sizes will show that actual theme counts cluster around 3-4 regardless of the stated maximum, because the LLM's own judgment dominates. Alternatively, the LLM may occasionally produce 7+ themes, revealing the cap is not enforced programmatically.

**If confirmed:** Either (a) remove the guideline entirely and let the LLM decide, or (b) add programmatic post-processing that actually enforces the cap by merging smallest themes.

#### Theory C: The real problem is the merging strategy, not the count (architecture is the bottleneck)

Merging excess themes into "Other" is the worst possible merge strategy. It creates a grab-bag category that no user will voluntarily explore. The issue is not "how many themes?" but "what happens to themes that don't make the cut?" A better architecture would use hierarchical themes (top-level themes with sub-themes) or present a ranked list where the user can expand past the default view.

**Prediction:** In experiments where "Other" contains 5+ parameters, those parameters will represent genuinely distinct research areas that got orphaned, not noise. Manual review of "Other" buckets will show they contain high-impact parameters that should have been in their own theme.

**If confirmed:** Replace the flat cap + "Other" merge with a two-tier system: show top N themes by default, offer "Show M more themes" as an expansion. No information is lost; cognitive load is managed through progressive disclosure rather than deletion.

### Methodology

This is purely analytical. No parameter sweep is needed.

1. **Enumerate representative scopes:** Identify 5 synthetic scope sizes (5 files / ~10 params, 10 files / ~25 params, 20 files / ~60 params, 30 files / ~100 params, 50 files / ~200 params)
2. **Reason about theme distribution:** For each scope size, estimate how many natural themes exist based on the nerd plugin's own file structure (agents, commands, configs, pipelines)
3. **Evaluate the "Other" bucket:** For each scope size, estimate what percentage of parameters would land in "Other" under the current cap of 6
4. **Test the LLM compliance hypothesis:** Check whether any programmatic enforcement exists (grep for post-processing of theme count). If there is none, the cap is purely advisory.
5. **Assess alternative models:** Compare flat cap, sqrt-scaled cap, and progressive disclosure

### Acceptance Criteria

| Criterion | Theory Tested | Pass Condition |
|-----------|---------------|----------------|
| "Other" bucket analysis | A | Estimate whether >30% of parameters in large scopes land in "Other" |
| LLM compliance check | B | Determine if theme count is enforced programmatically or only in prompt |
| "Other" quality audit | C | Assess whether merged parameters represent distinct research areas |
| Scaling formula viability | A | Proposed formula produces reasonable theme counts across scope sizes |

### Recommendation

**Change the cap to a scaled formula and replace "Other" with progressive disclosure.**

Specific edits to `agents/context-scanner.md`:

Replace lines 100-105:
```markdown
### Theme Size Guidelines

- **Minimum 2 themes** — even small scopes benefit from at least two distinct research threads
- **Default visible: min(ceil(sqrt(parameter_count)), 7)** — scales with scope size, never exceeds cognitive load limits
- **No hard maximum** — additional themes beyond the visible default are listed under "Additional themes (expand to view)" rather than merged into "Other"
- **Single-parameter themes** should be merged into the nearest related theme
- **Target density: 3-15 parameters per theme** — themes outside this range should be split or merged
```

Rationale: This preserves cognitive load management (sqrt scaling keeps visible themes in the 2-7 range for typical scopes) while eliminating information loss from the "Other" bucket. The LLM can produce as many themes as the data warrants; the UI concern is handled by progressive disclosure, not by deleting themes.

## E018: Eval Batch Size Table (currently 8GB=16, 16GB=64, 32GB=128, 64GB+=256)

**File:** `commands/nerd-setup.md:113`
**Current value:** Hardcoded lookup table mapping total system memory to eval batch sizes
**Role:** Determines the batch size used for evaluation during LLM training. Too large causes Metal OOM; too small wastes time on small batches.

### How It Works in Context

During `/nerd-setup`, the system detects total memory via `sysctl -n hw.memorysize`, then looks up the appropriate eval batch size from a fixed table. The comment on line 109 notes "Metal max buffer size is ~75% of total memory." The batch size is written to `hardware-profile.yaml` and used by `train.py` for all subsequent experiments.

The table makes several assumptions:
1. Total system memory is the right input (not available memory or Metal-specific limits)
2. The relationship between memory and safe batch size is a step function (not continuous)
3. The values 16, 64, 128, 256 are correct for their respective tiers
4. The tiers themselves (8/16/32/64) align with actual Apple Silicon configurations

### Analysis Against Apple Silicon Metal Memory Limits

Apple Silicon uses unified memory architecture (UMA). Key constraints:

| Chip | Configs Available | Metal Max Buffer | Metal Recommended Working Set |
|------|------------------|-----------------|------------------------------|
| M1 | 8GB, 16GB | ~75% of total | ~65% of total |
| M1 Pro | 16GB, 32GB | ~75% of total | ~65% of total |
| M1 Max | 32GB, 64GB | ~75% of total | ~65% of total |
| M1 Ultra | 64GB, 128GB | ~75% of total | ~65% of total |
| M2 | 8GB, 16GB, 24GB | ~75% of total | ~65% of total |
| M2 Pro | 16GB, 32GB | ~75% of total | ~65% of total |
| M3 | 8GB, 16GB, 24GB | ~75% of total | ~65% of total |
| M3 Pro | 18GB, 36GB | ~75% of total | ~65% of total |
| M4 | 16GB, 24GB, 32GB | ~75% of total | ~65% of total |
| M4 Pro | 24GB, 48GB | ~75% of total | ~65% of total |
| M4 Max | 36GB, 64GB, 128GB | ~75% of total | ~65% of total |

Problems with the current table:

1. **Missing tiers:** 24GB (M2/M3/M4), 36GB (M3 Pro, M4 Max), 48GB (M4 Pro), 128GB (M1 Ultra, M4 Max) are not represented. A 24GB machine falls into the 16GB tier (batch 64) when it could safely handle more.
2. **Step function creates waste:** A 24GB machine is 50% more capable than 16GB but gets the same batch size. A 36GB machine is 12.5% more than 32GB but could handle batch 192.
3. **No 128GB tier:** The Ultra configurations are lumped with 64GB.
4. **The 75% figure is approximate:** The actual Metal max buffer varies by macOS version and system load. Using a fixed percentage is fragile.

### Competing Theories

#### Theory A: The table needs more tiers and a continuous formula (parameter tuning)

The step function with only 4 tiers is too coarse for the current Apple Silicon lineup, which has 10+ distinct memory configurations. A continuous formula would eliminate the need to maintain a lookup table.

**Prediction:** Computing `batch_size = floor((memory_gb * 0.65) / bytes_per_sample)` with `bytes_per_sample` derived from model depth will produce batch sizes within 10% of empirically safe values for all Apple Silicon configs. The lookup table will be shown to under-provision by >50% for 24GB and 36GB machines.

**If confirmed:** Replace the lookup table with: `eval_batch_size = 2^floor(log2(memory_gb * 4))` (gives 32 for 8GB, 64 for 16GB, 64 for 24GB, 128 for 32GB, 128 for 36GB, 128 for 48GB, 256 for 64GB, 512 for 128GB). Calibration run validates the computed value.

#### Theory B: The table is moot because calibration should determine batch size empirically (feature is unnecessary)

Step 3 of nerd-setup already runs a calibration benchmark. The table is used to set an initial value before calibration, but if calibration binary-searches for the largest batch size that does not OOM, the table becomes a seed value at best. The real question is whether calibration is robust enough to find the right batch size on its own.

**Prediction:** If a binary-search calibration is implemented (start at 512, halve on OOM, double on success), it will converge to the correct batch size in 3-5 iterations (under 2 minutes) regardless of the initial seed. The lookup table provides zero information that calibration cannot discover.

**If confirmed:** Replace the table with a calibration-first approach: start at a generous default (256), binary-search downward on OOM. Record the empirically discovered max batch size. The table becomes dead code.

#### Theory C: The batch size does not significantly affect experiment throughput (metric is wrong)

Eval batch size affects only the evaluation phase, not training. If evaluation is <10% of total experiment time, then even a 4x difference in batch size might only change total runtime by 2-3%. The parameter feels important but may be dominated by training step time.

**Prediction:** Timing breakdown from calibration will show that evaluation accounts for <15% of total experiment time, making batch size optimization a micro-optimization with diminishing returns. The difference between batch_size=64 and batch_size=256 will be <30 seconds per experiment.

**If confirmed:** Set a conservative default (64), skip the table entirely, and document that batch size tuning is low-priority. Focus setup time on other calibration aspects.

### Methodology

1. **Tier gap analysis:** Enumerate all current Apple Silicon memory configs and identify which fall between existing tiers
2. **Formula derivation:** Compute a continuous formula from the 75% Metal limit and model memory requirements
3. **Timing proportion analysis:** From the hardware profile schema (lines 219-224 of nerd-setup.md), determine what fraction of `total_seconds` is evaluation vs training
4. **Binary search feasibility:** Estimate how many iterations a binary search calibration would need

### Acceptance Criteria

| Criterion | Theory Tested | Pass Condition |
|-----------|---------------|----------------|
| Tier coverage | A | Count of Apple Silicon configs that fall between current tiers |
| Formula accuracy | A | Proposed formula matches current table values +/-25% for existing tiers |
| Eval time proportion | C | Determine if eval is <15% of total experiment time from profile schema |
| Calibration convergence | B | Binary search from 512 converges in <5 iterations |

### Recommendation

**Replace the lookup table with a formula + calibration validation.**

Replace lines 112-117 of `commands/nerd-setup.md`:

```markdown
| Total Memory | Computed Eval Batch Size | Formula |
|-------------|-------------------------|---------|
| Any | `2^floor(log2(memory_gb * 4))` | Continuous, no gaps |

After computing the initial batch size, the calibration run (Step 3) validates it by attempting a single eval pass. If it OOMs, halve the batch size and retry. This eliminates the need to maintain a hardware-specific lookup table and automatically adapts to new Apple Silicon configurations.

Examples: 8GB -> 32, 16GB -> 64, 24GB -> 64, 32GB -> 128, 36GB -> 128, 48GB -> 128, 64GB -> 256, 128GB -> 512
```

Rationale: The continuous formula covers all current and future Apple Silicon configs without maintenance. The calibration validation catches cases where the formula overestimates (e.g., heavy background memory pressure). This is strictly better than the current 4-tier table.

## E019: Experiments Per Hour Estimates (currently M1=3-4, M1 Pro=6-8, etc.)

**File:** `commands/nerd-setup.md:37`
**Current value:** Hardcoded performance estimates per chip family
**Role:** Sets user expectations before calibration runs. Displayed in the hardware classification table.

### How It Works in Context

The table on lines 36-42 maps hardware to expected experiments/hour. This is used for:
1. **Pre-calibration expectation setting** -- tells users what to expect before the benchmark runs
2. **Hardware classification** -- determines which training variant to install
3. **Scheduling estimates** -- may influence how `/nerd-schedule` allocates time windows

After calibration (Step 3), the actual measured rate is written to `hardware-profile.yaml` as `experiments_per_hour: 3.1`. The pre-calibration estimate is never referenced again.

### Should E019 Be Removed Entirely?

The case for removal:

1. **The estimates are stale the moment they ship.** Performance depends on macOS version, background processes, model architecture changes in autoresearch-mlx, and thermal throttling. A table written in 2026 will be wrong by 2027.
2. **Calibration replaces them immediately.** Step 3 runs a real benchmark and produces the authoritative number. The estimates are displayed for approximately 60-120 seconds before being superseded.
3. **They create false expectations.** If the estimate says "6-8/hr" but calibration produces "4.2/hr", the user may think something is wrong with their setup. The estimate becomes a source of confusion rather than comfort.
4. **They require maintenance.** Every new Apple Silicon generation (M5, M6) requires updating this table, and the values are always guesses until someone benchmarks.

The case for keeping them:

1. **Users want to know what they are getting into.** A blank table with "will be determined by calibration" is less informative than a rough estimate.
2. **Hardware purchase decisions.** A user deciding between M4 Pro and M4 Max might consult this table.
3. **Sanity checking calibration.** If calibration says 0.5/hr on an M4 Max, the estimate table helps the user realize something is wrong.

### Competing Theories

#### Theory A: Replace estimates with ranges derived from calibration data (parameter tuning)

The current point estimates ("3-4", "6-8") look precise but are fabricated. Better to show ranges derived from actual calibration runs across hardware, with a clear "your mileage may vary" caveat.

**Prediction:** Collecting calibration results from even 3-5 different machines per chip family would produce ranges that differ from the current estimates by >30% for at least half the chip families.

**If confirmed:** Replace hardcoded estimates with a community-sourced range, or add a mechanism to report calibration results back to a shared dataset.

#### Theory B: Remove estimates entirely and make calibration mandatory (feature is unnecessary)

The estimates serve no functional purpose after calibration runs. They exist for a 60-second window of user comfort. The cost of maintaining them (stale data, false expectations, new-chip updates) exceeds the benefit.

**Prediction:** Removing the "Expected Performance" column and replacing it with "Run calibration to determine" will not increase user confusion or setup abandonment. Users who reach `/nerd-setup` are already committed to running the tool.

**If confirmed:** Remove the "Expected Performance" column from the hardware table. Add a single sentence: "Calibration (Step 3) will benchmark your specific machine." Keep the hardware-to-variant mapping, which is the functional part of the table.

#### Theory C: The estimates should be replaced by a model, not a table (model is wrong)

Instead of per-chip estimates, derive performance from measurable hardware characteristics: memory bandwidth (GB/s), GPU core count, and memory size. These are detectable at setup time and would produce estimates that automatically adapt to new chips.

**Prediction:** A simple linear model `experiments_per_hour = a * memory_bandwidth_gbps + b * gpu_cores + c` fitted to a handful of known datapoints will predict within 20% of actual calibration results for unseen hardware.

**If confirmed:** Replace the lookup table with a formula-based estimate. The formula degrades gracefully for new hardware (new chips just have different bandwidth/core counts, which are detectable).

### Methodology

1. **Usage analysis:** Determine exactly where the estimates are consumed (display only? scheduling? decision logic?)
2. **Staleness audit:** Compare the estimates to the one known calibration result in the profile schema (M1 Pro 16GB: 3.1 experiments/hr vs table estimate of 6-8)
3. **Maintenance cost:** Count how many chip variants exist that are not in the table
4. **Information value:** Assess whether the 60-second window of pre-calibration display justifies maintaining the table

### Critical Observation

The example hardware profile on line 219 shows an M1 Pro 16GB producing `experiments_per_hour: 3.1`. The table on line 37 estimates M1 Pro/Max (32GB+) at "6-8 experiments/hour." Even accounting for the 32GB+ qualifier, this is a 2x discrepancy that would confuse a 16GB M1 Pro user. The estimates are already demonstrably wrong for at least one configuration.

### Acceptance Criteria

| Criterion | Theory Tested | Pass Condition |
|-----------|---------------|----------------|
| Estimate vs calibration discrepancy | A, B | The example profile (3.1/hr) vs table estimate (6-8/hr) is a >40% mismatch |
| Post-calibration usage | B | Confirm estimates are never referenced after calibration writes to profile |
| Missing chip coverage | A, C | Count Apple Silicon variants not in the table |
| Model feasibility | C | Determine if memory bandwidth and core count are detectable at setup time |

### Recommendation

**Remove the "Expected Performance" column. Make calibration the single source of truth.**

Replace lines 35-42 of `commands/nerd-setup.md`:

```markdown
| Hardware | Training Variant |
|----------|-----------------|
| Apple Silicon (any) | autoresearch-mlx |
| NVIDIA GPU (24GB+) | autoresearch (original) |
| No GPU / CPU only | Skip LLM training |

Exact performance varies by chip, memory, and system load. Step 3 (calibration) will benchmark your machine and record the authoritative experiments-per-hour rate.
```

Rationale: The estimates are already wrong (3.1 actual vs 6-8 estimated for M1 Pro), they require per-chip maintenance for every new generation, and they are superseded within 60 seconds by calibration. The training variant selection is the only functional part of the table and does not require per-chip performance estimates. Removing them eliminates a maintenance burden and a source of user confusion.

## E020: DAG Theory Matching Rule (currently 1 file AND 1 tag)

**File:** `agents/plan-reviewer.md:33`
**Current value:** Two theories match if they share at least 1 identical `source_file` path AND at least 1 identical tag string
**Role:** Prevents re-testing hypotheses that have already been resolved. When the plan-reviewer generates new theories, it checks whether equivalent theories exist in the DAG.

### How It Works in Context

The matching rule (line 33) is a conjunction: `match = (shared_files >= 1) AND (shared_tags >= 1)`. This is used in Step 1.5 of the plan-reviewer to look up prior theories. The result determines whether to skip (SUPPORTED), exclude (REFUTED), or prioritize (INCONCLUSIVE) a theory.

The rule is evaluated by the LLM, not programmatically. The plan-reviewer reads the rule as an instruction and applies its judgment.

### Information Retrieval Analysis: Precision vs Recall

This is fundamentally an information retrieval problem. The matching rule determines which prior theories are "retrieved" as relevant to the current experiment. We can analyze it along precision/recall axes:

**Current rule: 1 file AND 1 tag (conjunction)**

- **Precision:** HIGH. Requiring both file overlap and tag overlap means matches are almost certainly relevant. A theory about "batch size tuning" in `nerd-setup.md` will only match another theory that also touches `nerd-setup.md` AND also uses a "batch size" or "tuning" tag.
- **Recall:** LOW. Misses theories that address the same concept in different files (e.g., batch size logic refactored from `nerd-setup.md` to `calibration.md`). Misses theories with synonymous but non-identical tags ("batch_size" vs "eval_batch" vs "memory_scaling").

**Alternative rules and their characteristics:**

| Rule | Precision | Recall | Risk |
|------|-----------|--------|------|
| 1 file AND 1 tag (current) | High | Low | Misses relevant refactored/renamed theories |
| 1 file OR 1 tag | Low | High | Floods reviewer with false matches |
| 2 files AND 1 tag | Very high | Very low | Almost never matches; DAG becomes useless |
| 1 file AND 2 tags | Very high | Low | Stricter than current; worse recall |
| 1 tag only | Low | High | Tags are often generic ("parameter_tuning"); too many matches |
| 1 file only | Medium | Medium | File overlap is strong signal but some false positives |
| Weighted score: `0.6*file_overlap + 0.4*tag_overlap >= threshold` | Tunable | Tunable | Adds complexity; threshold becomes a new parameter to tune |

### Competing Theories

#### Theory A: The conjunction is too strict; OR would be better (parameter tuning)

The AND rule sacrifices recall for precision. In a growing DAG with dozens of theories, missing relevant prior work (low recall) is more costly than surfacing a few irrelevant matches (low precision). The reviewer LLM can easily dismiss false positives, but it cannot discover theories it was never shown.

**Prediction:** As the DAG grows past 20 theories, the AND rule will fail to match theories that a human reviewer would consider relevant. Specifically, theories about the same conceptual parameter that were filed against different source files (due to refactoring) will be invisible to the current rule.

**If confirmed:** Switch to OR with a confidence qualifier: "Two theories match if they share at least 1 identical source_file path OR at least 1 identical tag string. For OR-only matches (file but no tag, or tag but no file), flag as 'possible match -- verify relevance.'"

#### Theory B: String-identical matching is the real problem, not the conjunction (model is wrong)

The rule requires "identical" source_file paths and "identical" tag strings. This is brittle. Files get renamed (`nerd-setup.md` -> `setup.md`). Tags are free-text and vary by author ("batch_size" vs "eval_batch_size" vs "memory_management"). The matching model should use semantic similarity, not string equality.

**Prediction:** Within the first 10 experiments producing DAG theories, at least 3 pairs of theories that a human would consider related will fail to match due to non-identical but synonymous tags or renamed files.

**If confirmed:** Replace string-identical matching with a two-tier approach:
- Tier 1 (exact): Current rule, for high-confidence matches
- Tier 2 (fuzzy): File basename match (ignoring directory) OR tag substring containment, flagged as "possible match"

#### Theory C: The DAG is too small for matching to matter yet (data is the bottleneck)

The nerd plugin is new. The DAG has zero theories. Matching rules are premature optimization. The rule will not be exercised meaningfully until there are 20+ theories across 5+ experiments. By the time matching matters, the requirements will be clearer from actual usage patterns.

**Prediction:** The first 5 experiments will produce <15 theories total. At this scale, the plan-reviewer could simply list ALL prior theories and let the LLM decide relevance. No matching rule is needed until the DAG is too large to show in full.

**If confirmed:** Replace the matching rule with a size-based strategy:
- DAG < 30 theories: Show all prior theories to the reviewer (no matching needed)
- DAG 30-100 theories: Use file OR tag matching
- DAG > 100 theories: Use file AND tag matching (precision matters at scale)

#### Theory D: LLM-evaluated rules are inherently unreliable (architecture is the bottleneck)

The matching rule is an instruction to an LLM, not programmatic code. The LLM may interpret "identical" loosely (matching "nerd-setup.md" to "commands/nerd-setup.md") or strictly (requiring exact path match). The rule's behavior depends on which model evaluates it, the prompt context length, and stochastic variation. No deterministic guarantees are possible.

**Prediction:** Given the same DAG and the same new theory, running the plan-reviewer 5 times will produce different sets of matched prior theories at least once, because the LLM's interpretation of "identical" varies.

**If confirmed:** Move matching out of the LLM and into a deterministic pre-processing step. Before the plan-reviewer runs, a script computes candidate matches using exact string comparison and passes only the matched theories as context. The LLM then evaluates relevance from the pre-filtered set, but does not perform the matching itself.

### Methodology

1. **Precision/recall analysis:** For each candidate rule (AND, OR, weighted), enumerate match behavior on synthetic theory pairs
2. **Brittleness audit:** Identify how file renames and tag synonyms break the current rule
3. **Scale analysis:** Estimate DAG growth rate (theories per experiment) and when matching becomes necessary
4. **Determinism test:** Assess whether LLM-evaluated matching produces consistent results

### Acceptance Criteria

| Criterion | Theory Tested | Pass Condition |
|-----------|---------------|----------------|
| Recall gap examples | A | Identify 2+ plausible theory pairs that AND-matching would miss |
| Synonym brittleness | B | Find tag variations in the codebase that would prevent identical matches |
| DAG growth estimate | C | Estimate experiments needed before DAG exceeds 30 theories |
| LLM consistency | D | Determine if matching is programmatic or purely LLM-interpreted |

### Recommendation

**Adopt a tiered strategy: show-all for small DAGs, OR-matching with confidence flags for larger ones, and move matching to a deterministic pre-processing step.**

Replace line 33 of `agents/plan-reviewer.md`:

```markdown
**Matching rule (tiered by DAG size):**

- **DAG < 30 theories:** Show ALL prior theories to the reviewer. No filtering needed.
- **DAG 30+ theories:** Pre-filter using: two theories are candidates if they share at least 1 source_file path (basename match, ignoring directory prefix) OR at least 1 tag (case-insensitive substring containment). Candidate matches are presented to the reviewer with a confidence flag:
  - **Strong match:** Both file AND tag overlap
  - **Possible match:** File OR tag overlap only -- reviewer should verify relevance

This matching should be performed by a deterministic pre-processing step (script or structured comparison), not left to LLM interpretation.
```

Rationale: The current AND rule is too strict for a young DAG and too brittle for string-identical matching. The tiered approach starts simple (show everything), scales to filtered matching as the DAG grows, and uses fuzzy matching (basename, substring) to handle renames and synonyms. Moving matching to a deterministic step eliminates LLM interpretation variance.

## Cross-Experiment Interactions

These four parameters interact in subtle ways:

1. **E017 (theme count) x E019 (performance estimates):** Both are "user comfort" parameters -- they shape the user's experience but don't affect functional correctness. Both could be replaced by adaptive/dynamic approaches.

2. **E018 (batch size table) x E019 (performance estimates):** Both are hardware-dependent lookup tables that calibration supersedes. If calibration is the source of truth, both tables are redundant.

3. **E020 (DAG matching) x E017 (theme count):** The context-scanner produces themes that eventually become DAG theories. If themes are poorly merged (E017's "Other" problem), the DAG will contain low-quality theories that pollute matching (E020).

4. **Shared pattern:** All four parameters are hardcoded heuristics that could be replaced by adaptive mechanisms. The meta-question is: should nerd have ANY hardcoded heuristics, or should everything be calibrated/learned?
