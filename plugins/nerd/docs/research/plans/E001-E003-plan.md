# E001-E003: Loop Control Parameter Analysis

## Overview

Three parameters in `commands/nerd-loop.md` collectively govern the nerd loop's exploration/exploitation tradeoff, total runtime budget, and self-awareness frequency. Because these are instruction parameters in a markdown-based agent plugin (not compiled code), "sweeping" means analytical reasoning from first principles and optimization literature, not empirical benchmarking.

This plan treats the three parameters as a coupled system and produces reasoned recommendations for each.

## DAG Context

No prior theories exist. This is the first research pass on loop control parameters.

## Parameters Under Study

### E001: Consecutive Discards Pivot Threshold
- **Location:** `commands/nerd-loop.md:184`
- **Current value:** 5
- **Role:** After N consecutive failed iterations within a single strategic phase, the loop pivots to a fundamentally different approach.
- **Range under consideration:** 3-10

### E002: Total Local Maximum Detection Depth
- **Location:** `commands/nerd-loop.md:195`
- **Current value:** 3 phases (normal -> pivot -> escalate)
- **Role:** Number of strategic phase transitions before declaring local maximum. Total budget = depth x E001 consecutive discards.
- **Range under consideration:** 2-5 phases

### E003: Loop Reflection Interval
- **Location:** `commands/nerd-loop.md:207`
- **Current value:** Every 5 iterations
- **Role:** How often the agent re-reads its full experiment log to find patterns and adjust strategy.
- **Range under consideration:** 3-15

### Interaction Model

These parameters form a budget equation:

```
Total worst-case iterations before termination = E002 x E001
  Currently: 3 x 5 = 15

Total reflections before termination = floor(E002 x E001 / E003)
  Currently: floor(15 / 5) = 3
```

The budget also has a cost dimension:
- Each iteration costs ~2-10 minutes wall time + API tokens
- At 5 min/iteration average, 15 iterations = ~75 minutes of fruitless searching before termination
- At 10 min/iteration, that becomes 2.5 hours

## Competing Theories

### E001: Consecutive Discards Pivot Threshold

#### Theory E001-A: The threshold is too generous (should be 3-4)
**Claim:** Five consecutive failures within a single strategic approach is too many. By iteration 3-4, the agent has exhausted the low-hanging fruit of its current approach. Iterations 4-5 are wasted tokens.

**Analogies:**
- **Early stopping in ML training:** Patience of 3-5 epochs is standard, but "epochs" in ML contain thousands of gradient updates. A single nerd iteration is more like a full training run with a different architecture -- making patience=3 more appropriate.
- **Simulated annealing:** At a given temperature, the acceptance probability drops exponentially. Analogously, the probability that attempt 5 succeeds where attempts 1-4 failed is low if the agent is truly stuck.
- **Genetic algorithms:** Tournament selection typically uses tournament sizes of 2-4. Larger tournaments increase selection pressure but reduce diversity. Five attempts at the same approach is a large tournament.

**Prediction:** If 3 is better, we would expect that in real loop runs, iterations 4-5 within a phase almost never produce improvements. The agent's "creative exhaustion" within an approach happens faster than 5 tries.

**If confirmed:** Reduce to 3. This also reduces total worst-case budget from 15 to 9 (with E002=3), saving ~30 minutes of fruitless computation.

#### Theory E001-B: The threshold is too aggressive (should be 7-10)
**Claim:** LLM agents have high variance in their ability to generate novel ideas. Unlike a gradient descent optimizer that systematically explores nearby points, an LLM might generate a mediocre idea on attempt 2, then a brilliant one on attempt 7. Five is not enough runway for the stochastic creativity process.

**Analogies:**
- **Random restarts in optimization:** The probability of finding a better local optimum increases with more restarts at the same temperature before cooling. If each "attempt" within a phase is a random restart, more is better.
- **Brainstorming research:** The "first ideas are obvious" principle -- in creative problem-solving, the best ideas often come after the obvious ones are exhausted (attempts 1-3), during the "forced creativity" phase (attempts 5-10).
- **Monte Carlo tree search:** Wider exploration at each node improves value estimates. The nerd loop is essentially MCTS where each phase is a node.

**Prediction:** If 7-10 is better, we would see that improvements sometimes come on iterations 6-10 within a phase, and that premature pivoting (at 3-5) causes the agent to abandon promising directions.

**If confirmed:** Increase to 7-8. Accept the higher cost per phase as a tradeoff for deeper exploration. Total budget becomes 21-24 iterations.

#### Theory E001-C: A fixed threshold is the wrong model entirely
**Claim:** The optimal pivot point depends on the problem domain, not on a fixed constant. For a well-understood optimization (tweaking numeric parameters), 3 attempts is plenty. For a creative restructuring task (redesigning an algorithm), 10 attempts might be too few. A fixed threshold cannot serve both.

**Analogies:**
- **Adaptive learning rates:** Modern optimizers (Adam, AdaGrad) adjust their step size based on gradient history. A fixed step size is known to be suboptimal.
- **Simulated annealing with adaptive cooling:** The Lam-Delosme schedule adjusts cooling rate based on acceptance ratio. Fixed cooling schedules are provably suboptimal for unknown landscapes.
- **Bandit algorithms (UCB, Thompson Sampling):** These dynamically balance exploration/exploitation based on observed variance, not fixed thresholds.

**Prediction:** If this theory is correct, we would see that the optimal E001 value varies significantly across different research focuses. Search relevance optimization might need 3, while algorithm redesign might need 8.

**If confirmed:** Replace the fixed threshold with an adaptive rule. For example: "Pivot when the last N attempts all achieved < X% of the best-to-baseline improvement range" or "Pivot when estimated marginal value of another attempt < cost." This is a more complex change to the instruction set.

### E002: Total Local Maximum Detection Depth

#### Theory E002-A: Three phases is correct (the Goldilocks theory)
**Claim:** The 3-phase structure (normal -> pivot -> escalate) maps well to a natural hierarchy of creative problem-solving: (1) exploit current understanding, (2) try a different approach, (3) challenge assumptions. This mirrors how human researchers work.

**Analogies:**
- **Multi-start optimization:** Three random restarts from different initial points is a common default. It balances thoroughness with efficiency.
- **Design thinking:** The "double diamond" model has explore/define/develop/deliver -- roughly 3 creative transitions.
- **Genetic algorithm generations:** Convergence typically happens within 3-5 generations for small populations. With population=1 (single agent), 3 generations is a reasonable default.

**Prediction:** If 3 is optimal, then adding a 4th or 5th phase would produce diminishing returns -- the agent would struggle to generate genuinely novel approaches beyond 3 strategic pivots.

**If confirmed:** Keep at 3. The current design is sound.

#### Theory E002-B: Two phases is sufficient
**Claim:** The distinction between "pivot" and "escalate" is semantic, not functional. An LLM agent told to "try something fundamentally different" and one told to "try the opposite of what's been working" will generate similar outputs. The escalation instruction doesn't meaningfully expand the search space beyond what pivoting already does.

**Analogies:**
- **Restart strategies:** In SAT solving, Luby restarts (geometric restart schedule) are often as effective as more complex strategies. Simplicity wins.
- **Occam's razor for meta-parameters:** Every additional phase is a meta-parameter that must be justified. The burden of proof is on complexity.

**Prediction:** If 2 phases is sufficient, then the "escalate" phase would produce changes qualitatively similar to the "pivot" phase. The agent can't meaningfully distinguish between "different approach" and "opposite approach."

**If confirmed:** Reduce to 2 phases (normal -> pivot -> local_maximum). Total budget drops from 15 to 10 (with E001=5), saving ~25 minutes.

#### Theory E002-C: Depth should scale with problem complexity, not be fixed
**Claim:** Simple optimizations (parameter tuning, constant adjustment) hit their ceiling in 1-2 phases. Complex optimizations (algorithm design, architectural changes) need 4-5 phases because the search space is larger and the agent needs more strategic resets to explore it.

**Analogies:**
- **Iterative deepening depth-first search (IDDFS):** The optimal depth limit depends on the solution depth, which is unknown a priori. IDDFS solves this by progressively deepening.
- **Simulated annealing temperature schedule:** Longer cooling schedules are needed for more complex energy landscapes. The number of temperature steps should scale with problem dimensionality.

**Prediction:** If this is correct, a fixed depth of 3 would over-invest in simple problems and under-invest in complex ones. We would see simple optimizations plateau by phase 2, while complex ones would still be finding improvements in phase 3.

**If confirmed:** Make depth configurable or adaptive. Simple approach: allow `/nerd-loop` to accept a `--depth` parameter. Adaptive approach: continue adding phases as long as each phase produces at least one improvement.

#### Theory E002-D: The phase structure itself is wrong -- use continuous temperature
**Claim:** Discrete phases (normal/pivot/escalate) create artificial boundaries. A better model is a continuous "creativity temperature" that rises as consecutive failures accumulate, rather than discrete jumps at fixed thresholds.

**Analogies:**
- **Simulated annealing (literal):** Temperature is continuous, not stepped. The acceptance probability changes smoothly.
- **Curriculum learning:** Difficulty increases gradually, not in discrete jumps.
- **Epsilon-greedy with decay:** Exploration probability decays smoothly, not in steps.

**Prediction:** If this is correct, we would see that the agent's behavior doesn't meaningfully change at phase boundaries -- it either over-pivots (too dramatic a change) or under-pivots (ignores the instruction to change approach). A continuous temperature instruction like "Your last N attempts failed. The more failures, the more radically different your next attempt should be" would produce smoother exploration.

**If confirmed:** Replace discrete phases with a continuous instruction that scales creativity pressure with consecutive failure count. This is a significant rewrite of the loop control logic.

### E003: Loop Reflection Interval

#### Theory E003-A: 5 iterations is too infrequent (should be 3)
**Claim:** Pattern recognition is most valuable early, when the agent is forming its mental model of the optimization landscape. Waiting 5 iterations means the agent may repeat similar failed approaches before recognizing they share a common flaw. More frequent reflection catches dead-end patterns sooner.

**Analogies:**
- **Online learning:** Lower batch sizes (more frequent updates) converge faster in non-stationary environments. The optimization landscape the agent faces is effectively non-stationary because its own changes alter it.
- **Retrospectives in agile:** Sprint retrospectives every 1-2 weeks outperform monthly ones for small teams. With 5-10 minute iterations, every 3 iterations is a 15-30 minute "sprint."

**Prediction:** If 3 is better, we would see the agent making similar mistakes across iterations 1-5 that it would have caught with earlier reflection.

**If confirmed:** Reduce to 3. Cost: slightly more tokens per reflection (re-reading the log) but likely offset by fewer wasted iterations.

#### Theory E003-B: 5 is correct, or reflection should be less frequent (7-10)
**Claim:** Reflection is expensive -- re-reading the full experiment log consumes context window and tokens. More importantly, premature pattern-finding can lead to false generalizations ("architecture changes don't work" after seeing 2 fail, when the 3rd would succeed). The agent needs enough data points before drawing conclusions.

**Analogies:**
- **Statistical significance:** You need N >= 5-10 observations to draw reliable conclusions about a distribution. Reflecting at 3 iterations means drawing conclusions from 3 data points.
- **Exploration in reinforcement learning:** Too-frequent policy updates cause instability. The agent needs time to explore before updating its strategy.

**Prediction:** If less frequent is better, we would see that reflections at interval=3 produce premature generalizations that constrain the search space too early.

**If confirmed:** Keep at 5 or increase to 7. Accept the risk of repeated mistakes as the cost of avoiding premature convergence.

#### Theory E003-C: Reflection interval should be tied to phase transitions, not iteration count
**Claim:** The most valuable time to reflect is at strategic inflection points (when entering a new phase after a pivot), not on a fixed schedule. Reflecting mid-phase is less useful because the agent is executing a single strategy. Reflecting at phase boundaries helps the agent synthesize what it learned from the previous phase and plan the next one.

**Analogies:**
- **Meta-learning:** MAML reflects at the end of each task, not mid-task. The "task" here is a phase.
- **Chapter summaries:** You summarize at the end of a chapter, not every 5 pages.

**Prediction:** If this is correct, phase-boundary reflection would produce more actionable insights than fixed-interval reflection.

**If confirmed:** Change reflection from "every N iterations" to "at the start of each new phase + every N iterations within a phase." This gives both structured and periodic reflection.

#### Theory E003-D: Reflection is unnecessary -- the agent's context window already handles it
**Claim:** The LLM agent naturally has access to its recent history through the conversation context. Explicit "re-read the log" instructions are redundant because the agent already "remembers" what it tried. Reflection adds overhead without meaningful benefit.

**Analogies:**
- **Transformer attention:** The model already attends to all previous tokens in context. Explicit summarization is only needed when context overflows.
- **Working memory vs. long-term memory:** Within a single session, the agent's working memory (context window) already contains the experiment log.

**Prediction:** If reflection is unnecessary, removing the explicit reflection step would have no measurable impact on loop performance -- the agent would still avoid repeating failed approaches and build on successes.

**If confirmed:** Remove the explicit reflection step. Rely on the natural context window. Add reflection only when the context window is at risk of overflow (very long loops).

## Interaction Analysis

### The Budget Triangle

The three parameters form a "budget triangle" where any two constrain the third:

```
Total iterations before termination = E001 x E002
Reflections per run = floor(E001 x E002 / E003)
Cost = (E001 x E002) x (avg_iteration_cost)
```

Current values: 5 x 3 / 5 = 15 iterations, 3 reflections, ~75-150 minutes at plateau.

| E001 | E002 | E003 | Total Iters | Reflections | Est. Cost (min) |
|------|------|------|-------------|-------------|-----------------|
| 3    | 2    | 3    | 6           | 2           | 30-60           |
| 3    | 3    | 3    | 9           | 3           | 45-90           |
| 5    | 3    | 5    | 15          | 3           | 75-150          |
| 5    | 4    | 5    | 20          | 4           | 100-200         |
| 7    | 3    | 5    | 21          | 4           | 105-210         |
| 7    | 5    | 7    | 35          | 5           | 175-350         |
| 10   | 5    | 10   | 50          | 5           | 250-500         |

The current 15-iteration budget sits in a reasonable middle ground. Going below 9 risks premature termination. Going above 25 risks excessive cost for diminishing returns.

### Correlation Effects

1. **E001 and E002 are multiplicative:** Increasing both compounds the budget. If E001=7 and E002=5, total budget is 35 iterations -- potentially 3-6 hours of fruitless searching. This is almost certainly too expensive.

2. **E003 should scale with E001:** If the pivot threshold is 5, reflecting every 5 iterations means exactly one reflection per phase. If the pivot threshold is 3, reflecting every 5 means some phases get no reflection at all. The ratio E003/E001 matters:
   - E003/E001 < 1: Multiple reflections per phase (potentially excessive)
   - E003/E001 = 1: One reflection per phase (current state)
   - E003/E001 > 1: Some phases get no reflection (information loss)

3. **E002 determines strategic diversity:** More phases require the agent to generate more fundamentally different approaches. With depth=5, the agent needs 5 distinct strategic frameworks. LLMs may not have that many meaningfully different approaches for a given problem.

### The Exploration/Exploitation Spectrum

Mapping the parameter space to optimization theory:

| Configuration | Exploration/Exploitation | Analogy |
|--------------|--------------------------|---------|
| E001=3, E002=2 | Heavy exploitation, light exploration | Greedy search with 1 restart |
| E001=5, E002=3 | Balanced (current) | Simulated annealing with moderate patience |
| E001=7, E002=4 | Heavy exploration | Multi-start with deep search per start |
| E001=10, E002=5 | Maximum exploration | Exhaustive search (expensive) |

## Literature-Grounded Analysis

### Simulated Annealing Patience

In simulated annealing, the number of iterations at each temperature (the "epoch length" or "Markov chain length") determines how thoroughly the algorithm explores each energy level before cooling. Key findings:

- **Hajek's theorem (1988):** For guaranteed convergence, the cooling schedule must be logarithmic -- extremely slow. Practical implementations use faster schedules and accept approximate solutions.
- **Practical guidance (Kirkpatrick et al.):** Epoch length of 5-20 at each temperature is typical for moderate-dimensional problems. The nerd loop's E001=5 falls at the lower end of this range.
- **Adaptive epoch lengths (Lam & Delosme, 1988):** Adjusting epoch length based on acceptance ratio outperforms fixed lengths. This supports Theory E001-C.

**Implication for E001:** The current value of 5 is defensible as a lower-bound practical choice. Increasing to 7 would be more conservative but also more expensive. An adaptive approach would be ideal but adds instruction complexity.

### Genetic Algorithm Generations

In genetic algorithms, the number of generations before convergence depends on population size and selection pressure:

- **Small populations (1-5):** Converge quickly (5-20 generations) but to potentially suboptimal solutions.
- **The nerd loop has population=1** (single agent), making it more like a (1+1)-ES (evolution strategy). For (1+1)-ES, convergence on unimodal functions takes O(n log n) steps where n is dimensionality. For multimodal functions, random restarts are essential.

**Implication for E002:** The 3-phase structure acts as 3 random restarts of a (1+1)-ES. The optimization literature suggests 3-5 restarts is reasonable for moderate landscapes. The current E002=3 is at the low end but defensible for cost reasons.

### Early Stopping in ML Training

Early stopping uses a "patience" parameter: how many epochs without improvement before stopping. Standard values:

- **Small datasets:** Patience 3-5
- **Large datasets:** Patience 10-20
- **Key insight:** Patience should scale with the noise level. High-variance metrics need more patience to avoid stopping on a noise spike.

**Implication for E001:** LLM-generated code changes have extremely high variance (much higher than gradient updates). This argues for higher patience (7-10). However, each "epoch" in the nerd loop is enormously more expensive than a training epoch, which argues for lower patience (3-5). These forces roughly cancel, supporting the current E001=5.

### Reflection and Meta-Learning

- **MAML (Model-Agnostic Meta-Learning):** Performs meta-updates after completing entire tasks, not mid-task. This supports Theory E003-C (phase-boundary reflection).
- **Experience replay (DQN):** Replays past experiences at every step. This supports more frequent reflection.
- **The compromise:** In practice, periodic reflection (every K steps) is a standard approach in many RL algorithms. K=5 is not unusual.

**Implication for E003:** The current interval of 5 is defensible. The strongest argument for change is Theory E003-C: aligning reflection with phase transitions rather than a fixed interval.

## Recommendations

### E001: Consecutive Discards Pivot Threshold

**Recommendation: Keep at 5 (no change).**

**Rationale:**
- Theory E001-A (reduce to 3-4) is plausible but risky. Three attempts may not be enough for an LLM to exhaust a strategic direction, especially for complex optimization targets. The cost savings (2 fewer iterations per phase) are modest.
- Theory E001-B (increase to 7-10) is also plausible but the cost argument works against it. Each additional attempt within a dead-end phase is pure waste.
- Theory E001-C (adaptive) is the most intellectually compelling but adds significant instruction complexity for uncertain benefit.
- The simulated annealing literature puts 5 at the lower end of practical epoch lengths, and early stopping literature suggests patience=5 is standard. The current value sits at a defensible equilibrium.
- **Strong signal to change:** If future loop runs consistently show that improvements in iterations 4-5 of a phase are near-zero, reduce to 3. If improvements at iterations 4-5 are common, increase to 7.

**Future validation:** Add logging to track which iteration within a phase produced each improvement. After 5+ loop runs, analyze the distribution.

### E002: Total Local Maximum Detection Depth

**Recommendation: Keep at 3 phases (no change).**

**Rationale:**
- Theory E002-A (3 is correct) has the strongest theoretical support. Three phases map to a natural hierarchy: exploit -> explore nearby -> explore far. This mirrors multi-start optimization with 3 restarts.
- Theory E002-B (reduce to 2) is tempting for cost savings but risks premature termination. The "escalate" phase serves a distinct function: challenging assumptions rather than just trying different approaches. Whether an LLM can meaningfully distinguish these is an empirical question.
- Theory E002-C (scale with complexity) is sound but adds UX complexity.
- Theory E002-D (continuous temperature) is the most interesting theoretically but would require a significant rewrite of the loop control instructions. It should be considered for v2 if the current discrete model proves inadequate.
- **Strong signal to change:** If the escalate phase (phase 3) never produces improvements across multiple loop runs, reduce to 2. If it frequently does, the current value is validated.

**Future validation:** Track improvements per phase. If phase 3 improvements are < 10% of total improvements across 5+ runs, reduce to 2.

### E003: Loop Reflection Interval

**Recommendation: Reduce from 5 to 3, and add phase-boundary reflection.**

**Rationale:**
- This is the parameter with the strongest case for change. The current E003/E001 ratio is 1.0, meaning exactly one reflection per phase. This is fragile -- if E001 changes, the reflection timing becomes misaligned.
- Theory E003-A (reduce to 3) is supported by the online learning analogy. More frequent updates converge faster in non-stationary environments. The cost per reflection (re-reading the log) is modest compared to the cost of wasted iterations.
- Theory E003-C (phase-boundary reflection) is the strongest theoretical argument. The most valuable time to reflect is at strategic inflection points. This can be combined with periodic reflection.
- Theory E003-D (reflection is unnecessary) is plausible within short loops but breaks down as context windows fill up in longer runs. Explicit reflection serves as a forcing function for the agent to synthesize patterns.

**Specific change to `commands/nerd-loop.md:204-208`:**

```markdown
7. REFLECT: At the start of each new phase (after a pivot or escalation), AND
   after every 3 iterations within a phase, re-read the experiment log.
   - What patterns emerge? (Do all architecture changes help? Do parameter tweaks plateau?)
   - What's the ratio of kept to discarded?
   - Are the improvements getting smaller? (diminishing returns)
   - (At phase boundaries only) What did the previous phase teach you that should
     inform the new approach?
```

This gives 1-2 reflections per phase (at 3 iterations and at phase entry), which is more than the current 1 reflection per phase, at modest additional cost.

## Ablation Baselines

For future empirical validation, the following ablation tests should be run when loop usage data is available:

1. **No pivot mechanism (E001=infinity):** The agent runs until E002 x E001 total failures without any strategic reset. Tests whether pivoting adds value.
2. **Single phase (E002=1):** The agent runs E001 iterations and terminates. Tests whether multi-phase exploration adds value.
3. **No reflection (E003=infinity):** The agent never explicitly re-reads the log. Tests whether reflection adds value vs. natural context retention.
4. **Maximum reflection (E003=1):** The agent reflects after every iteration. Tests whether there's a point of diminishing returns for reflection frequency.

## Review Notes

### What was considered and decided

1. **Parameter sweep approach rejected.** These are markdown instruction parameters for an AI agent, not numeric hyperparameters in code. There is no automated sweep mechanism. The methodology is analytical reasoning from optimization literature + future empirical validation.

2. **Interaction effects prioritized.** Rather than analyzing each parameter independently, the plan treats them as a coupled system with a budget equation. This prevents recommendations that are locally optimal but globally incoherent (e.g., increasing both E001 and E002, which would compound the budget).

3. **Conservative bias applied.** For E001 and E002, the recommendation is "no change" because the current values sit at defensible equilibrium points in the optimization literature. Changing them without empirical data risks moving away from a reasonable default. The burden of proof is on change.

4. **E003 is the highest-confidence change.** The case for reducing reflection interval and adding phase-boundary triggers is strongest because (a) it doesn't affect the budget equation, (b) it has clear theoretical support from meta-learning literature, and (c) the cost is minimal (slightly more log re-reading).

5. **Instrumentation is the real deliverable.** The most valuable outcome of this analysis is not the parameter changes themselves but the instrumentation plan. Without per-iteration and per-phase logging, all future parameter decisions will remain theoretical. Phase 2 (instrumentation) is more important than Phase 1 (the E003 change).

6. **Adaptive approaches deferred.** Several theories (E001-C, E002-C, E002-D) suggest replacing fixed parameters with adaptive mechanisms. These are intellectually compelling but add instruction complexity. They should be revisited after empirical data exists.

### Gaps acknowledged

- **No empirical validation possible yet.** All recommendations are theoretical. The plan includes a validation strategy but cannot execute it without loop run data.
- **LLM behavior is not well-modeled by optimization theory.** The analogies to simulated annealing, genetic algorithms, and early stopping are useful but imperfect. An LLM agent's "search process" is fundamentally different from numeric optimization -- it is more creative, more variable, and less predictable. The analogies provide useful intuitions but not precise predictions.
- **Token cost is unmodeled.** The plan acknowledges that each iteration has a cost but does not model it precisely. The actual cost depends on context window size, log length, and the specific metric command. A full cost model would require empirical measurement.
